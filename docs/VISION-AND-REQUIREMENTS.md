## 2. 성공 정의 (한 문장 ~ 두 문장)

### 2.1 플랫폼·기술 병목 (핵심)

- **“어떤 툴을 켜도 동일한 `get_context`(맥락 패키지)가 주입된다.”**  
→ Cursor·클로드 등에 갇힌 **대화 메모·선호·스택 정보**가 분산되지 않도록 하는 **기술적 성공 조건**으로 적합하다.

### 2.2 업무·의사결정 연속성 (보조 한 줄, 선택)

- **“요청이 어디서 오든, 목표·제약·최근 결정이 빠지지 않고, 계획·생성·평가 하네스 안에서만 SoT에 반영된다.”**  
→ 복잡 작업 **오케스트레이션** 병목과 **맥락 유지**를 함께 명시할 때 사용한다.

### 2.3 전 세계적 병목과의 관계

- **동일 `get_context`**는 “플랫폼에 갇힌 맥락”이라는 **보편 병목**을 겨냥한 조건이다.  
- “전 세계” 서술까지 포함하려면 **인간 측 성과(의사결정 연속성)**를 위 **2.2** 한 줄을 더하는 것이 자연스럽다.

## 3. 핵심 병목 (사용자 정의)

### 3.1 병목 1 — 공유되지 않는 메모리·맥락

- 커서·클로드 등 **플랫폼별 대화 메모**와 **선호**(스택, 패딩 등)가 **서비스마다 분리**됨.
- **목표**: 명령·리퀘스트를 어디서 보내든, **어떤 에이전트가 처리하든** “내가 원하는 것·작업 맥락”이 **동일 레이어**로 공유됨.

**대응 방향 (원칙)**

- **대화 로그 전체 동기화**는 목표에서 제외하거나 후순위.
- **프로필·금지·선호·활성 프로젝트·결정·용어** 등 **재사용 가능한 구조화 정보**를 SoT에 둔다.
- **MCP 서버**로 Cursor·클로드·(추후) ChatGPT·제미나이 등이 **같은 맥락·지침**을 읽게 한다.

### 3.2 병목 2 — 오케스트레이션

- 복잡한 작업 요청 시 **어느 에이전트·역할이 처리할지** 라우팅·분해가 어렵다.

**대응 방향 (원칙)**

- **Planner → Generator → Evaluator(P/G/E)** 파이프라인으로 **반영 전 검증**.
- 초기에는 **태그·역할 힌트** 수준으로 시작하고, **LLM 라우터·멀티에이전트**는 데이터·하네스가 쌓인 뒤 고도화.

## 4. 산업·방법론 맥락 (비전과의 정렬)

- **프롬프트 엔지니어링 → 컨텍스트 엔지니어링 → 하네스 엔지니어링**으로의 흐름과 정렬.
- **온톨로지·그래프 RAG·메타온톨로지**로 “세상에 대한 문맥과 눈치”를 쌓는다는 비전과 정렬.
- 구현 순서: **구조화된 스냅샷 → 벡터 RAG → 지식 그래프 → 메타온톨로지(개념 버전·출처 신뢰도·규칙 계층)** 점진 도입.

## 5. 기능 요구사항 (원문 요구 + 합의)

### 5.1 수집 및 변환 (Data Ingestion)

- **입력**: 유튜브, RSS, 인스타·스레드(링크·스크린샷), 스크린샷 일반, 깃허브 소스 등.
- **처리**: LLM을 통해 **사용자 정의 마크다운(MD) 규격**으로 파싱·변환·메타데이터 부여.
- **소스별 현실**:
  - **RSS·유튜브(자막/메타)·GitHub**는 정책·API 범위에서 우선 구현하기 좋음.
  - **인스타·스레드**: 공식 API 한계 가능. **권장 전제** — (1) 공개 URL이 열리면 본문/메타 시도, (2) **스크린샷 + OCR**, (3) 링크 복사·큐. 사용자 워크플로: **스크린샷·링크 복사 후 정제**.

### 5.2 유니버셜 메모리 (Hybrid Memory)

- 단순 텍스트만이 아니라 **지식 그래프 + 시맨틱 캐시**로 개인 데이터와 **최신 맥락**을 보존.
- **단계**: 파일·풀텍스트 검색 → 벡터 인덱스 → 엔티티·용어 링크 → 그래프·메타온톨로지 본격화.
- **시맨틱 캐시**: “최근 N결정·프로젝트 컨텍스트” 등 **상한 있는 스냅샷**부터 정의 가능.

### 5.3 오케스트레이션 (Rulebook & Skills)

- **룰북·스킬·프로젝트·태스크** 관리.
- **AI가 작업 전 지침을 읽도록 강제** — MCP `get_context` / 세션 게이트 / P/G/E와 연계.
- **구조화를 Skills보다 먼저** (폴더·스키마·MCP 계약 → Cursor/에이전트 Skills).

### 5.4 통합 동기화 (MCP Server)

- Cursor·클로드·ChatGPT·제미나이 등 **툴이 달라도 동일 맥락·지침** 유지.
- 예시 도구: `get_context`, `append_decision`, `log_evaluation`(Evaluator 판정 구조화 저장), (선택) `search_memory`, `plan_task`, `evaluate_artifact` 등 — 설계 시 확정.

### 5.5 단일 앱(OS)

- 인제스천·메모리·룰·프로젝트·태스크 쪼개기·액션을 **하나의 앱**으로 통합하는 목표.
- UI는 초기에 **CLI + MCP만**으로도 비전 검증 가능; 데스크톱/웹은 단계적.

## 6. 아키텍처 원칙 (합의)

### 6.1 진실의 원천(SoT)

- **SoT**: “이 값이 맞다”고 정한 **단일 원본**. 에이전트 런타임의 **최종 진실**은 **자체 앱 + 저장소**(로컬 Git 우선, 필요 시 DB·Supabase 등).
- **완전 로컬 필수는 아님**. 다만 **에이전트가 믿는 최종본이 어디인지**는 한 축으로 고정.
- **암호화된 클라우드 백업**: 노트북 분실 등에 대비한 **잠긴 복사본**; 평문 무작정 업로드와 구분.

### 6.2 노션

> ⚠️ 2026-06 현행화: ADR-006으로 **노션 = 지식·기획 정본(SSoT)** 확정 — §0.3 참조. 아래는 v1 기록.

- **필수는 아님**. 기획·태스크·DB UI·사람이 보기 좋은 정리에 유리.
- **노션 AI 존재**만으로 **무조건 쌍방 동기가 정답은 아님**. 충돌 시 **어느 쪽이 최종인지**가 모호해짐.
- **권장 패턴**: **자체 SoT = 최종**, 노션은 **미러·입력 큐·뷰**; 노션 AI 출력도 **SoT 반영 시 P/G/E** 경로 권장.

### 6.3 옵시디언

- 현재 미사용; **필수 아님**. 필요 시 **Vault = SoT 폴더와 동일**하게 두는 방식만 검토.

### 6.4 깃허브 연동

- 코드·이슈·PR 맥락이 핵심이면 **중요도 높음**; 노트만이면 선택. 개발 중심 OS면 **조기 연동 가치**.

## 7. 사용자 워크플로 (대화에서 확정한 흐름)

- 유튜브·RSS·스레드/인스타에서 정보 발견 → **스크린샷 또는 링크 복사**.
- **클로드로 정제·검증** 후 **보관·정리** — 원래는 노션 DB를 염두에 둠.
- **합의**: 동일 파이프를 **자체 앱**으로 구현 가능. 흐름은 **캡처 큐 → 정제(LLM) → 사용자 규격 MD → SoT 저장 → (선택) 노션 푸시**.

## 8. 정제(정제 카드)·스크린샷·노션 (세부 합의)

### 8.1 정제의 정의

- **정제** = **적용·재사용 가능**한 상태로 만들고, **AI와 사용자 검증 절차를 통과**한 것.
- **P/G/E** 중 **Evaluator + 사람 게이트**와 동일 축.

### 8.2 정제 성공 기준 (사용자 확정)

- **스크린샷을 다시 열지 않고 카드만 보고 업무에 쓸 수 있으면** 목적 달성. 그래서 정제한다.
- **원본 출처·링크·(SoT 내) 파일 경로**는 가능하면 **항상 남김** — 카드는 일하고, 출처는 추적·검증용.

### 8.3 텍스트형 vs 그림형 스크린샷

- **원본 이미지 파일의 주 저장**: **SoT** (`ingest/raw/` 등).
- **노션**: **정제 카드** + 원본으로의 **링크/경로**; 필요 시 썸네일. 무거운 원본을 노션에만 두면 이전·용량 이슈.
- **정보 추출**(글·숫자)과 **시각 참고**(레이아웃·무드)는 **주제·분야마다 비중이 다름**; **한쪽만** 고정하지 않음. 카드에 **정보 축·시각 축**을 모두 둘 수 있음.

### 8.4 노션 DB에 남기는 것

- **주로 정제된 카드**(요약·태그·프로젝트 연결).
- **원본 클립**은 SoT에 두고 노션에는 **참조**가 일반적.

## 10. 디바이스·운영

- **멀티 PC**: “동시 편집” = 여러 기기에서 **같은 파일을 거의 동시에** 수정할 때의 충돌 이슈. 사용자는 **주로 노트북 단일** → 빈도 낮으면 Git + 습관으로 충분한 경우 많음.
- **모바일**: 나중 단계로 해도 됨.
- **셀프 호스팅**: **필수 아님**. 로컬 앱 + MCP로 v0 가능. MCP를 **상시 URL**로 쓰거나 신뢰 최소 클라우드가 필요할 때 VPS 등 검토.
- **월 비용 상한**: 한 달 API·호스팅·구독 **지출 한도**를 정하면 스택 선택이 쉬움 (0원 로컬부터 가능).

## 12. 개인 vs 향후 플랫폼

> ⚠️ 2026-06 현행화: 메인 트랙에 수익화(VHK 출시·판매 + 스튜디오 결제·론칭) 추가 — §0.2 참조.

- **현재**: 개인 용도.
- **향후**: 플랫폼화 가능성 — **데이터 스키마·내보내기·테넌트 분리 여지**만 초기부터 열어 두면 됨. 당장 멀티유저 구현은 필수 아님.

## 13. 메타온톨로지·“눈치” (사용자 응답 반영)

- **H14**: 코드 스타일만이 아니라 **전 영역**에서 맥락·눈치 목표.
- **H15**: 개념 버전·출처 신뢰도·규칙 계층 **모두 중요**. **구조화된 시스템**으로 AI 에이전트 통제, 업무 효율·자동화·워크플로 최적화·속도·**정확도에 근접**·**최신 유지**·**원클릭(딸깍) 간편성** — 목표로 명시.

**보정**: “정확도 100%”는 **원칙 목표**로 두고, 구현은 **Evaluator·인용·인간 승인**으로 근접.

## 15. Planner / Generator / Evaluator (초안 요약)

- **Planner**: 요청 → 목표·제약·태스크 분해·`context_refs`가 담긴 플랜 객체.
- **Generator**: 태스크별 산출물(MD·패치 등) + 메타(출처·신뢰도).
- **Evaluator**: 플랜·SoT 대비 검증 → pass / revise / reject; **pass 전 SoT 영구 반영 금지(또는 draft만)**.
