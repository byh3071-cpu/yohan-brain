## 프롬프트가 뭐길래

프롬프트(prompt)는 AI에게 보내는 지시문이에요. 식당에서 주문할 때를 생각해보세요. "뭐 맛있는 거 주세요"라고 하면 셰프가 알아서 아무거나 내놓겠죠? "매운 거 빼고, 해산물 파스타, 크림 소스로 해주세요"라고 하면 원하는 음식이 나올 확률이 훨씬 높아요.

AI도 똑같아요. 프롬프트가 모호하면 AI는 그럴듯하지만 원하지 않는 답을 줘요. 프롬프트가 구체적이면 원하는 답에 훨씬 가까워져요.

좋은 프롬프트에는 세 가지 축이 있어요.

#table(
  columns: (auto, 1fr, 1fr),
  fill: (x, y) => if y == 0 { practice-color.primary } else if calc.rem(y, 2) == 1 { practice-color.bg } else { color.white },
  stroke: 0.5pt + practice-color.border,
  inset: 10pt,
  table.header(
    text(fill: color.white, weight: "bold")[축],
    text(fill: color.white, weight: "bold")[설명],
    text(fill: color.white, weight: "bold")[식당 비유],
  ),
  text(weight: "bold", fill: practice-color.primary)[맥락(Context)], [상황, 배경, 관계 등 작업의 배경 정보를 제공], [알레르기, 취향, 예산, 몇 명인지를 알려주기],
  text(weight: "bold", fill: practice-color.primary)[제약조건(Constraints)], [지켜야 할 규칙, 포함/제외할 내용을 명시], [맵기 빼고, 해산물 위주로 해달라고 요청],
  text(weight: "bold", fill: practice-color.primary)[형식(Format)], [출력의 구조와 형태를 지정], [코스 요리인지, 단품인지 정하기],
)

이 세 축을 하나씩 추가하면서 결과가 어떻게 달라지는지 직접 확인해볼게요.

# 실습: 프롬프트 3단 변환
// ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

// ━━━ Step 1 ━━━
#practice-step(1, "나쁜 프롬프트로 시도하기")[
  먼저 가장 간단한 프롬프트로 시작해볼게요. ChatGPT 또는 Claude를 열고 아래 프롬프트를 그대로 입력해보세요.

  ```
  사과 메일 써줘
  ```

  AI가 뭔가를 써줄 거예요. 하지만 이 결과를 실제로 쓸 수 있을까요? 아마 여러 가지 문제가 보일 거예요.

  결과를 받으면 아래 항목을 체크해보세요.

  - 누구에게 보내는 메일인지 맞나요?
  - 어떤 상황의 사과인지 맞나요?
  - 메일의 톤이 적절한가요?
  - 바로 복사해서 보낼 수 있는 수준인가요?

  대부분의 경우 네 항목 중 하나도 만족하지 못할 거예요. AI는 아무 정보도 받지 못했으니 전부 추측으로 채울 수밖에 없거든요.

  결과를 메모장에 복사해두세요. 나중에 비교할 거예요.
]

// ━━━ Step 2 ━━━
#practice-step(2, "작업 맥락을 추가해보기")[
  이번에는 새 대화를 열고, 작업 맥락을 추가해볼게요.

  ```
  온라인 쇼핑몰의 CS팀장이 고객에게 보내는 사과 메일을 써줘.
  배송이 늦어진 상황이야.
  ```

  Step 1의 결과와 비교해보세요. 달라진 점이 보이나요?

  작업 맥락을 추가하면 AI는 상황에 맞는 관점에서 글을 써요. "CS팀장이 고객에게 보내는 사과 메일"이라는 맥락이 들어가면 AI는 누가 누구에게, 왜 쓰는 글인지를 이해하고 적절한 어투와 구조를 잡아줘요.

  하지만 여전히 부족한 부분이 있어요. **구체적으로 어떤 상황인지, 어떤 조건을 지켜야 하는지** AI는 모르거든요. 결과를 메모장에 복사해두세요.
]

// ━━━ Step 3 ━━━
#practice-step(3, "구체적 맥락 + 제약조건 + 형식을 전부 추가하기")[
  이번에는 다시 새 대화를 열고, 맥락 + 제약조건 + 형식을 모두 제공해볼게요.

  ```
  <context>
  발신: 온라인 쇼핑몰 CS팀장
  수신: 노트북을 주문한 고객 (이미 1회 항의 전화를 한 상태)
  사안: 배송 예정일보다 5일 지연, 원인은 물류 센터 시스템 오류
  </context>

  <task>
  위 상황에서 사과 + 보상안 메일을 작성해주세요.
  - 구체적인 보상안(15% 할인 쿠폰)을 포함해주세요
  </task>

  <format>
  - 제목 포함
  - 3단 구조: 사과 → 원인 설명 → 보상안
  - 300자 내외
  - 정중하되 과도하게 비굴하지 않은 톤
  </format>
  ```

  이 결과는 어떤가요? Step 1, Step 2와 비교해보세요.

  구체적인 맥락이 들어가면 AI는 더 이상 추측하지 않아요. 상황을 정확히 알고 있으니 구체적인 내용을 채울 수 있거든요. 형식이 들어가면 출력 구조까지 예측 가능해져요. XML 태그(`<context>`, `<task>`, `<format>`)로 영역을 분리하면 AI가 각 부분의 역할을 명확히 이해해요.

  > ***핵심****: 같은 AI, 같은 주제인데 프롬프트 구조만 바꿨을 뿐이에요. 그런데 결과는 완전히 달라요. 이것이 프롬프트 구조의 힘이에요.
]

// ━━━ Step 4 ━━━
#practice-step(4, "세 가지 결과를 나란히 비교하기")[
  메모장에 복사해둔 세 가지 결과를 나란히 놓고 비교해볼게요. 아래 표를 직접 채워보세요.

  #table(
    columns: (auto, 1fr, 1fr, 1fr),
    fill: (x, y) => if y == 0 { practice-color.primary } else if calc.rem(y, 2) == 1 { practice-color.bg } else { color.white },
    stroke: 0.5pt + practice-color.border,
    inset: 10pt,
    table.header(
      text(fill: color.white, weight: "bold")[평가 항목],
      text(fill: color.white, weight: "bold")[Step 1],
      text(fill: color.white, weight: "bold")[Step 2],
      text(fill: color.white, weight: "bold")[Step 3],
    ),
    text(weight: "bold", fill: practice-color.primary)[상황 정확도], [?/5], [?/5], [?/5],
    text(weight: "bold", fill: practice-color.primary)[톤 적절성], [?/5], [?/5], [?/5],
    text(weight: "bold", fill: practice-color.primary)[바로 사용 가능 여부], [?/5], [?/5], [?/5],
    text(weight: "bold", fill: practice-color.primary)[구조/형식 완성도], [?/5], [?/5], [?/5],
    text(weight: "bold", fill: practice-color.primary)[총점], [?/20], [?/20], [?/20],
  )

  점수를 매겨보면 패턴이 보여요. 작업 맥락만 추가해도 점수가 오르고, 구체적인 제약조건과 형식까지 추가하면 점수가 크게 뛰어요. 이게 "맥락 + 제약조건 + 형식"이라는 구조의 효과예요.

  여기서 중요한 점은 이거예요. AI가 똑똑해져서 결과가 좋아진 게 아니에요. **사람이 정보를 잘 전달해서** 결과가 좋아진 거예요. AI의 성능을 끌어올리는 가장 빠른 방법은 모델을 바꾸는 게 아니라 프롬프트를 바꾸는 거예요.
]

#try-it[
  아래 세 가지 상황 중 하나를 골라 "맥락 + 제약조건 + 형식" 구조로 프롬프트를 작성해보세요.

  **상황 A: 업무 메일 작성**
  - 맥락: 발신자, 수신자, 관계, 상황을 구체적으로 설명
  - 제약조건: 반드시 포함할 내용, 제외할 내용
  - 형식: 메일 제목 포함, 단락 수, 톤 지정

  **상황 B: 보고서 요약**
  - 맥락: 보고서의 주제, 핵심 수치, 독자가 누구인지
  - 제약조건: 원본에 있는 수치만 사용, 추측 금지
  - 형식: 3줄 요약, 핵심 수치 포함, 경어체

  **상황 C: 아이디어 브레인스토밍**
  - 맥락: 해결하려는 문제, 산업/분야, 현재 시도한 것
  - 제약조건: 예산 범위, 실현 가능한 수준
  - 형식: 번호 목록, 각 아이디어에 장단점 포함

  작성한 프롬프트를 AI에 입력하고, 결과를 확인해보세요. 마음에 들지 않는 부분이 있다면 맥락이나 형식을 수정해서 다시 시도해보세요.
]

# 자주 하는 실수
// ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

프롬프트를 처음 구조화할 때 자주 빠지는 함정이 있어요.

### 실수 1: 맥락 없이 막연하게 요청하기

"SNS 글 써줘"만으로는 부족해요. 어떤 제품인지, 타겟이 누구인지, 어떤 플랫폼에 올릴 건지, 톤은 어떤지를 알려줘야 해요. 작업의 구체적인 맥락이 없으면 AI는 모든 것을 추측으로 채울 수밖에 없어요.

### 실수 2: 한 번에 너무 많이 요청하기

"마케팅 전략 세우고, 예산도 잡고, 일정도 짜고, 보고서도 써줘." 이렇게 하면 모든 항목이 얕아져요. 한 번에 하나의 작업만 요청하고, 결과를 확인한 후 다음 작업을 요청하는 게 훨씬 효과적이에요.

### 실수 3: 형식 지정을 잊기

AI는 기본적으로 길게 쓰려는 경향이 있어요. "3줄로", "표로", "번호 목록으로" 같은 형식을 지정하지 않으면 불필요하게 긴 답변이 돌아와요. 형식을 지정하면 결과물의 활용도가 크게 올라가요.

### 실수 4: AI를 만능으로 기대하기

프롬프트를 아무리 잘 써도 AI는 최신 데이터를 모르거나, 사실을 지어내거나, 본인만의 내부 사정을 알 수 없어요. 프롬프트가 해결하는 것은 **전달의 문제**이지 **AI 능력의 한계**가 아니에요. 이 차이는 다음 차시에서 더 자세히 다룰 거예요.

// ━━━ 트러블슈팅 ━━━
#troubleshoot((
  (
    q: "맥락을 줬는데도 응답이 별로 달라지지 않아요.",
    a: "맥락이 아직 추상적일 수 있어요. '마케팅 글 써줘' 대신 '스타트업 B2B SaaS 제품의 인스타그램 캡션을 써줘. 타겟은 20대 개발자, 톤은 캐주얼'처럼 상황, 대상, 조건을 구체적으로 추가해보세요.",
  ),
  (
    q: "프롬프트를 길게 쓰면 AI가 지시를 무시해요.",
    a: "프롬프트가 너무 길면 AI가 일부를 놓칠 수 있어요. 핵심 지시는 프롬프트의 맨 앞이나 맨 뒤에 배치하세요. 중간에 있는 지시는 무시될 확률이 높아요.",
  ),
  (
    q: "같은 프롬프트를 넣었는데 매번 다른 답이 나와요.",
    a: "이건 정상이에요. AI는 확률적으로 텍스트를 생성하기 때문에 매번 조금씩 다른 결과가 나와요. 핵심 구조와 내용이 일관되면 정상적으로 작동하는 거예요.",
  ),
