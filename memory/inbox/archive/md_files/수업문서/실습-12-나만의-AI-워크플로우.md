# 0. 오늘 만들 결과물

이번 실습은 실습편 전체의 피날레예요. 지금까지 배운 모든 기술을 조합해서 여러분의 실제 반복 업무 1개를 AI로 자동화하는 워크플로우를 설계하고, 실행하고, 평가하고, 문서화하는 것이 목표예요. 이번 실습이 끝나면 여러분만의 AI 워크플로우 문서가 완성돼요.

#summary-box(title: "배경지식: 워크플로우 설계의 원칙")[
  AI 워크플로우는 단순히 "AI에게 일을 시키는 것"이 아니에요. 좋은 워크플로우는 세 가지 질문에 답할 수 있어야 해요.

  1. **무엇을 자동화하는가**: 반복적이고, 규칙이 명확하고, 실수 비용이 낮은 작업이 적합해요
  2. **어떤 도구를 조합하는가**: 프롬프트, CLAUDE.md, MCP, Skills 중 필요한 것을 선택해요
  3. **어떻게 검증하는가**: 자동화된 결과가 올바른지 확인하는 방법이 있어야 해요

  이 세 질문에 답하지 못하는 자동화는 오히려 위험해요. "AI가 잘못했는데 그걸 몰랐다"는 상황이 가장 나쁜 시나리오거든요.
]

### 예시: "주간 코드 리뷰 리포트" 워크플로우 설계

구체적인 예시로 설계 과정을 보여줄게요.

**업무**: 매주 금요일, 이번 주 커밋을 분석해서 팀 코드 리뷰 리포트를 작성하는 업무

==== 프롬프트 설계

```
Role: 시니어 개발자로서 코드 리뷰 보고서를 작성하세요
Context: 이번 주 (월~금) 커밋 기반, 팀 프로젝트
Task: 주요 변경사항 요약, 코드 품질 이슈, 개선 제안 도출
Format: 마크다운 리포트, 팀원별 섹션 구분
```

==== 컨텍스트 설계 (CLAUDE.md)

```markdown
## 주간 리뷰 규칙
- 리포트는 한글로 작성
- 팀 코딩 컨벤션: PEP 8 + 타입 힌트 필수
- 리뷰 관점: 가독성, 유지보수성, 보안
- 칭찬:지적 비율 = 2:1 이상
```

==== 도구 설계 (MCP)

- **파일시스템 MCP**: 프로젝트 코드 접근
- **GitHub MCP** (선택): PR 정보 접근

==== 절차 설계 (Skill)

```markdown
---
description: "이번 주 커밋을 분석하여 주간 코드 리뷰 리포트를 생성합니다"
---

# 주간 코드 리뷰 리포트

1. `git log --since="last monday" --oneline`로 이번 주 커밋 목록 확인
2. 각 커밋의 diff를 분석하여 주요 변경사항 파악
3. 팀원별로 변경사항 그룹화
4. 코드 품질 이슈 식별 (타입 힌트, 에러 처리, 테스트)
5. 개선 제안 작성
6. docs/weekly-review-YYYY-MM-DD.md에 리포트 저장
```

==== 검증 설계

- 리포트에 모든 팀원이 포함되었는지 확인
- 칭찬:지적 비율이 2:1 이상인지 확인
- 누락된 커밋이 없는지 git log와 대조

> ****핵심****: 워크플로우의 가치는 "한 번 잘 되는 것"이 아니라 "매번 안정적으로 되는 것"이에요. 재현성이 낮으면 자동화의 의미가 없어요. 재현성을 높이려면 스킬의 절차를 더 구체적으로 작성하세요.

#practice-step(4, "워크플로우를 문서화해서 누구나 사용할 수 있게 만들어요")[
  워크플로우가 동작하면 이제 문서로 정리할 차례예요. 잘 정리된 워크플로우 문서는 다른 사람(또는 미래의 나)이 바로 사용할 수 있게 해줘요.

  프로젝트 폴더에 워크플로우 문서를 만들어보세요.

  ```
  code ~/my-workflow/WORKFLOW.md
  ```

아래 템플릿을 사용해서 문서를 작성하세요.

- [ ] MCP 서버 등록: [필요한 MCP 목록]

- [ ] Skill 파일 작성 완료

## 실행 방법

## 구성 요소

// ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

이 흐름은 우연이 아니에요. 의도적으로 "기초 → 한계 인식 → 도구 → 조합"의 순서로 구성했어요.

### 기술 스택 전체도

```
여러분의 요청
    │
    ▼
┌─────────────────────────────────────┐
│  프롬프트 (Role + Context + Task)    │  ← 실습 1~3
│  + 제약 조건, 출력 형식             │
└─────────────┬───────────────────────┘
              │
              ▼
┌─────────────────────────────────────┐
│  CLAUDE.md (프로젝트 컨텍스트)       │  ← 실습 7
│  빌드, 스타일, 구조                 │
└─────────────┬───────────────────────┘
              │
              ▼
┌─────────────────────────────────────┐
│  Skills (절차적 지식)                │  ← 실습 9~10
│  /commit, /review, /deploy          │
└─────────────┬───────────────────────┘
              │
              ▼
┌─────────────────────────────────────┐
│  MCP (외부 도구)                     │  ← 실습 8
│  파일시스템, 메모리, API             │
└─────────────┬───────────────────────┘
              │
              ▼
┌─────────────────────────────────────┐
│  에이전트 루프 (자율 실행)           │  ← 실습 11
│  Gather → Act → Verify → 반복       │
└─────────────────────────────────────┘
```

각 계층이 아래 계층 위에 쌓이는 구조예요. 프롬프트 없이 MCP만 있으면 의미가 없고, CLAUDE.md 없이 에이전트 루프를 돌리면 일관성이 떨어져요. **전체가 조화를 이룰 때 진정한 가치가 나와요.**

> ****핵심****: AI 자동화의 핵심은 "하나의 완벽한 도구"가 아니라 "여러 도구의 조합"이에요. 프롬프트가 기초를 잡고, CLAUDE.md가 맥락을 주고, MCP가 능력을 확장하고, Skills가 절차를 정의하고, 에이전트 루프가 자율적으로 실행해요.

#summary-box(title: "실습편 전체 핵심 정리")[
  - **프롬프트**: AI와 소통하는 가장 기본적인 방법이에요. 구조화하면 품질이 올라가요
  - **CLAUDE.md**: 프로젝트 단위로 AI의 행동을 일관되게 제어해요
  - **MCP**: AI에게 외부 도구를 연결하는 표준 프로토콜이에요
  - **Skills**: 반복되는 작업 절차를 파일로 정의해서 한 번의 명령으로 실행해요
  - **에이전트 루프**: 분석 → 실행 → 검증을 자동으로 반복해요
  - **워크플로우**: 이 모든 것을 조합해서 실무 업무를 자동화해요
  - **검증은 필수*: 자동화 결과를 검증하는 방법이 없으면 자동화하지 마세요
]
