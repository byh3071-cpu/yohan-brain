# 파인튜닝: 대부분은 필요 없고, 나머지는 LoRA로 충분하다

*AI 도입 상담에서 가장 자주 듣는 말이 *우리 데이터로 모델을 학습시키고 싶다*이다. 이 요청의 90%는 파인튜닝이 아닌 다른 방법으로 해결된다. 진짜 파인튜닝이 필요한 10%에서도 모델 전체를 건드릴 일은 거의 없다.*

## 파인튜닝이 필요 없는 90%를 먼저 구분해야 한다

파인튜닝(Fine-Tuning)이란 사전학습된 모델의 가중치를 특정 목적에 맞게 추가 학습시키는 과정이다. 옷을 사서 체형에 맞게 수선하는 것과 같다. 기성복의 원단과 재단은 그대로 두고, 소매 길이와 허리 폭만 조정한다.

문제는 수선이 필요하지 않은 경우가 훨씬 많다는 점이다. 실전에서 *파인튜닝이 필요하다*고 판단되는 요청의 대부분은 지식 부족 문제다. 지식 부족은 파인튜닝이 아니라 RAG(Retrieval-Augmented Generation, 검색 증강 생성)로 해결한다. RAG는 외부 데이터베이스에서 필요한 정보를 가져와 모델에 전달하는 방식이다. 모델 자체를 바꾸지 않고 참고 자료를 건네주는 것이다.

판단 기준은 하나다. *지식이 부족한가, 행동을 바꿔야 하는가.*

| [*상황*], [*해법*], [*이유*],  [회사 내부 문서를 참조해 답하게 하고 싶다], [RAG], [지식 부족. 외부에서 가져오면 된다],
  [모델이 항상 우리 회사 톤으로 말하게 하고 싶다], [파인튜닝], [행동 변경. 모델 자체를 바꿔야 한다],
  [최신 법률 정보를 알게 하고 싶다], [RAG], [지식 부족. 법률 DB를 연결하면 된다],
  [의료 보고서를 특정 양식으로 출력하게 하고 싶다], [파인튜닝], [행동 변경. 출력 형식이 문제다],)

이 표의 핵심은 왼쪽 두 행과 오른쪽 두 행의 차이다. *우리 회사 데이터로 학습시켜야 한다*는 요청은 거의 예외 없이 RAG로 해결된다. 도서관에 책을 추가하는 것과 사서의 응대 방식을 바꾸는 것은 전혀 다른 문제다. 대부분의 기업이 원하는 것은 책 추가이지 사서 교체가 아니다.

> **핵심**: 파인튜닝은 모델의 *행동 패턴*을 바꿔야 할 때만 필요하다. 지식을 추가하고 싶다면 RAG가 답이다. 이 구분을 먼저 하면 불필요한 비용의 90%를 절약할 수 있다.

## 진짜 파인튜닝이 필요한 네 가지 경우

RAG로 해결되지 않는 10%가 있다. 모델의 행동 자체를 바꿔야 하는 경우다. 구체적으로 네 가지다.

첫째, *톤과 스타일을 바꿔야 할 때*다. 모델이 항상 한국어 존댓말로, 특정 브랜드의 문체로 답하게 하려면 프롬프트에 매번 규칙을 넣는 것보다 모델 자체에 학습시키는 것이 안정적이다. 배우에게 매 장면마다 대본을 읽어주는 것과, 배우가 캐릭터를 체화하는 것의 차이와 같다.

둘째, *도메인 전문 용어를 자연스럽게 쓰게 할 때*다. 의료, 법률, 금융 분야의 전문 용어는 일반 모델이 어색하게 처리하는 경우가 많다. 파인튜닝으로 해당 도메인의 언어 패턴을 학습시키면 자연스러운 전문 응답이 가능해진다.

셋째, *지연 시간이 중요해서 프롬프트를 줄여야 할 때*다. 프롬프트에 긴 규칙을 매번 넣으면 토큰 수가 늘어나고 응답 시간이 길어진다. 파인튜닝으로 규칙을 모델에 내재화하면 프롬프트가 짧아지고 비용도 줄어든다.

넷째, *특정 출력 형식을 일관되게 유지해야 할 때*다. 항상 JSON으로, 항상 특정 스키마로 출력하게 만드는 것은 프롬프트만으로는 신뢰도가 떨어진다. 수백 번 호출하면 한두 번은 형식이 깨진다. 파인튜닝으로 형식을 내재화하면 일관성이 크게 향상된다.

네 가지의 공통점이 있다. 전부 *형식(form)*의 문제라는 것이다. 파인튜닝은 형식을 가르치는 데 탁월하지만, 새로운 사실을 주입하는 데는 약하다. 이 구분을 놓치면 비용만 쓰고 효과를 보지 못한다.

## 전체 파인튜닝은 왜 실전에서 사라지고 있는가

파인튜닝이 필요하다고 결정됐다면, 다음 질문은 *어떤 방법으로*다. 가장 직관적 방법은 전체 파인튜닝(Full Fine-Tuning)이다. 모델의 모든 파라미터를 업데이트한다.

문제는 비용이다. 7B 파라미터 모델의 전체 파인튜닝에도 수십 GB의 GPU 메모리가 필요하다. 70B 모델이면 수백 GB다. 여러 장의 고가 GPU를 클러스터로 묶어야 하고, 비용은 수십만 원에서 수백만 원에 이른다. 더 큰 문제는 과적합(overfitting)이다. 소량의 데이터로 전체 파라미터를 수정하면, 모델이 학습 데이터에만 맞춰지고 일반 능력을 잃는다. 이것을 치명적 망각(Catastrophic Forgetting)이라 한다. 영어 원어민이 한국어를 집중 학습하다 영어 문법을 잊어버리는 것과 같다.

이 두 문제 때문에 전체 파인튜닝은 실전에서 빠르게 사라지고 있다. 대안이 있기 때문이다.

## LoRA: 전체의 1%만 바꿔도 95%의 효과를 낸다

LoRA(Low-Rank Adaptation, 저차 적응)는 2021년 Hu 등이 제안한 방법이다. 핵심 아이디어가 명쾌하다. 모델의 거대한 가중치 행렬을 직접 수정하지 않고, 작은 *어댑터* 행렬 두 개를 옆에 붙여서 학습시킨다. 원본 가중치는 얼린 채로 두고, 어댑터만 업데이트한다.

비유하면 이렇다. 전체 파인튜닝은 건물 전체를 리모델링하는 것이다. LoRA는 건물은 그대로 두고 각 방에 작은 가구만 추가하는 것이다. 건물의 구조는 바뀌지 않지만, 방의 기능은 달라진다.

| [*항목*], [*전체 파인튜닝*], [*LoRA*],  [수정 파라미터], [100%], [0.1~1%],
  [GPU 메모리], [수십~수백 GB], [단일 GPU 가능],
  [비용], [수십만~수백만 원], [수만 원],
  [성능], [100%], [90~95%],
  [과적합 위험], [높음], [낮음],)

이 표에서 핵심은 성능 대비 비용이다. 파라미터의 1%만 수정해도 전체 파인튜닝 성능의 90~95%에 도달한다. 나머지 5%를 위해 비용을 10배 이상 쓸 이유가 없다. Hu 등의 원 논문이 이 비율을 GPT-3 175B에서 실증했고, 이후 오픈소스 커뮤니티에서 7B부터 65B까지 다양한 모델에서 재현됐다.

LoRA의 또 다른 장점은 *모듈성*이다. 어댑터는 원본 모델과 분리되어 있으므로, 용도별로 여러 어댑터를 만들어 교체할 수 있다. 하나의 기본 모델에 법률용 어댑터, 의료용 어댑터, 고객 응대용 어댑터를 각각 연결하는 식이다. 스마트폰에 케이스를 바꿔 끼우는 것처럼 용도에 따라 전환할 수 있다.

### QLoRA: 더 적은 자원으로 더 큰 모델을

QLoRA(Quantized LoRA)는 LoRA에 4비트 양자화(Quantization)를 결합한 기법이다. 양자화란 모델의 숫자 표현 정밀도를 낮추는 것으로, 32비트 실수를 4비트 정수로 압축하면 메모리 사용량이 약 8분의 1로 줄어든다. Dettmers 등이 2023년에 발표한 이 기법 덕분에 65B 파라미터 모델을 단일 48GB GPU에서 파인튜닝하는 것이 가능해졌다.

QLoRA의 등장이 파인튜닝의 진입 장벽을 극적으로 낮췄다. 이전에는 대기업이나 연구소에서만 가능했던 대형 모델 파인튜닝이 클라우드 GPU 한 장으로 가능해진 것이다. 오픈소스 파인튜닝 커뮤니티의 폭발적 성장은 이 기술이 직접적 원인이다.

> **핵심**: LoRA는 모델의 0.1~1%만 수정해 전체 파인튜닝 성능의 90~95%를 달성한다. QLoRA는 여기에 양자화를 더해 단일 GPU로 65B 모델 파인튜닝을 가능하게 만들었다. 이 두 기법이 2026년 파인튜닝의 사실상 표준이다.

## 정렬 파인튜닝: DPO와 RLHF 사이의 실전 선택

앞 장에서 RLHF와 DPO의 원리를 다뤘다. 여기서는 실전에서 어떤 기준으로 선택하는지를 본다.

핵심 질문은 *위험 수준*과 *자원*이다. 의료 AI를 만들고 있다면 RLHF가 맞다. 표현력이 높고, 보상 모델을 세밀하게 조정할 수 있다. 고객 챗봇의 톤을 맞추는 것이라면 DPO가 효율적이다. 빠르고 저렴하며 안정적이다.

아래 그림은 파인튜닝 의사결정의 전체 흐름을 보여준다. 첫 번째 분기점에서 RAG로 해결 가능한지를 확인하고, 그렇지 않은 경우에만 파인튜닝으로 진행한다.

![파인튜닝 의사결정 흐름. RAG로 해결 가능한 경우가 첫 번째 분기이며, 대부분의 요청이 여기서 걸러진다.](/assets/consulting-cropped/13-finetune.png)

이 그림에서 주목할 것은 첫 번째 분기점이다. 모델의 행동을 바꿔야 하는지를 먼저 묻고, RAG로 가능한 경우를 먼저 거른다. 이 단계를 건너뛰면 불필요한 파인튜닝 비용이 발생한다.

실전에서 많이 쓰이는 조합은 LoRA + DPO다. LoRA로 도메인 적응을 하고, DPO로 출력 톤과 거부 정책을 다듬는다. GPU 한 장에서 며칠이면 끝난다. 2026년 현재 정렬 기법 없이 LLM을 프로덕션에 배포하는 기업은 거의 없지만, 정렬의 구현 비용은 2년 전 대비 크게 낮아졌다.

## 2026년의 새로운 선택지

파인튜닝 기법은 계속 진화하고 있다. 두 가지가 주목할 만하다.

첫째, GRPO(Group Relative Policy Optimization, 그룹 상대 정책 최적화)다. 앞 장에서 DeepSeek R1의 학습 기법으로 소개했다. 같은 질문에 여러 답변을 생성하고 그룹 내 상대 순위로 학습한다. 보상 모델도 가치 모델도 필요 없다. 수학과 코드처럼 정답이 명확한 영역에서 특히 효과적이다. 올림픽 체조에서 심사위원 대신 선수끼리 서로를 평가하는 방식으로, 객관적 기준이 있는 종목에서만 작동한다.

둘째, KTO(Kahneman-Tversky Optimization)다. 이름에서 알 수 있듯 행동경제학의 전망 이론에서 착안했다. DPO는 *A가 B보다 좋다*라는 쌍 비교 데이터가 필요하다. KTO는 *A는 좋다* 또는 *A는 나쁘다*라는 단일 판단만으로 충분하다. 데이터 수집이 훨씬 쉬워진다. 사용자가 답변에 좋아요/싫어요만 누르면 학습 데이터가 된다. 별점 5점 척도로 세밀하게 평가하는 것보다 수집 비용이 극적으로 낮다.

기법의 수가 늘어난 것은 좋은 신호다. 2022년에는 RLHF 하나뿐이었다. 2026년에는 과제의 성격, 데이터의 종류, 자원의 규모에 따라 최적의 조합을 고를 수 있다. 선택지가 많아진 만큼, 어떤 경우에 어떤 기법을 쓸지를 판단하는 능력이 더 중요해졌다.

다음 장에서는 이 정렬 기법들의 근본적 한계와, 그 한계를 넘으려는 시도들을 다룬다. 인간의 피드백 없이도 AI를 정렬할 수 있는가라는 질문이다.
