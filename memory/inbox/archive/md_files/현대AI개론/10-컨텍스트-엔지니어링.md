# 컨텍스트 엔지니어링: 정보 건축의 시대

*프롬프트만으로는 끝났다면, 대체 뭘 해야 하는가. 답은 컨텍스트 윈도우 전체를 설계하는 것이다. 어떤 정보를, 어떤 형태로, 어떤 순서로, 얼마나 넣을 것인가. 이 설계 행위에 2025년 6월, 이름이 붙었다. 컨텍스트 엔지니어링.*

## 프롬프트를 넘어서면 정보 건축이 보인다

앞 장에서 프롬프트가 컨텍스트 윈도우의 0.25%에 불과하다는 사실을 확인했다. 나머지 99.75%를 채우는 것이 진짜 작업이라면, 그 작업에는 어떤 원칙이 필요한가.

컨텍스트 엔지니어링(Context Engineering)은 LLM 추론 시점에 모델이 보는 토큰 집합 전체를 큐레이션하고 유지하는 전략의 집합이다. 시스템 프롬프트, 도구 정의, MCP, 검색 결과, 메모리, 메시지 이력, 상태를 포함한다. Andrej Karpathy가 2025년 6월 25일에 내린 정의가 이 분야의 출발점이 됐다. *다음 단계를 위해 컨텍스트 윈도우를 딱 알맞은 정보로 채우는 섬세한 예술이자 과학*이라는 것이다.

이 정의에서 빠뜨리면 안 되는 단어가 두 개 있다. *딱 맞는 정보만(just the right)*과 *바로 다음 한 단계를 위해(for the next step)*다.

첫째, 많이 넣는다고 좋지 않다. Chroma Research(Kelly Hong 외, 2025)가 18개 SOTA 모델을 벤치마크한 결과, 컨텍스트가 논리적으로 일관될수록 오히려 성능이 나빠졌다. 200K 토큰 모델이 50K 토큰 지점에서 이미 불균일한 성능 붕괴를 보였다. 윈도우의 25%만 채웠는데도 무너진 셈이다.

![컨텍스트 윈도우의 구성. 시스템 프롬프트, 도구 정의, 검색 문서, 대화 기록, 도구 출력의 5개 영역으로 나뉘며 도구 출력이 80%+를 차지한다.](../../../02-context-window.png)

둘째, 컨텍스트는 정적이 아니라 동적이다. 프롬프트는 대화 시작 전에 한 번 쓰고 끝이다. 컨텍스트 엔지니어링에서는 매 스텝마다 윈도우 내용이 바뀐다. 버그를 찾을 때는 에러 로그와 관련 파일을 로드하고, 원인을 찾았으면 에러 로그를 빼고 수정 대상 파일만 유지하고, 테스트를 돌릴 때는 수정 파일과 테스트 파일만 남긴다. 이전 스텝에서 필요했던 정보가 다음 스텝에서는 노이즈가 되기 때문이다.

## 프롬프트 엔지니어링과 무엇이 다른가

Harrison Chase(LangChain)가 두 개념의 관계를 정리했다. 프롬프트 엔지니어링은 한 문장의 문구를 다듬는 공예이고, 컨텍스트 엔지니어링은 윈도우에 무엇이 어떤 형식으로 언제 들어가는지의 정보 건축(information architecture)이다. 관계는 포함이다. 프롬프트 엔지니어링은 컨텍스트 엔지니어링의 부분 집합이다.

건물을 짓는 것에 비유하면 이해가 빠르다. 프롬프트를 쓰는 것은 벽에 페인트칠을 하는 것이다. 컨텍스트를 설계하는 것은 건물의 골조를 세우는 것이다. 구조 설계가 4축 프레임워크이고, 자재 선정이 어떤 소스를 넣을 것인가이며, 동선 계획이 정보 배치 순서(캐시 효율)이고, 하중 계산이 토큰 예산 관리(70~80%)이며, 내진 설계가 실패 모드 방어다. 페인트칠도 중요하지만, 골조가 부실하면 건물이 무너진다.

| 항목 | 프롬프트 엔지니어링 | 컨텍스트 엔지니어링 |
| --- | --- | --- |
| 대상 | 한 턴의 입력 문구 | 윈도우 전체의 토큰 구성 |
| 시점 | 대화 시작 전 1회 | 매 스텝마다 동적 갱신 |
| 범위 | 단어 선택, 예시, 역할 | 시스템 프롬프트 + 도구 + 메모리 + 이력 + 도구 출력 |
| 비유 | 벽 페인트칠 | 건물 골조 |
| 관계 | 부분 집합 | 상위 집합 |

## 컨텍스트 윈도우 안에 실제로 들어 있는 것

많은 사람이 컨텍스트를 자신이 쓴 메시지로만 생각한다. 실제로 컨텍스트 윈도우를 열어보면 다섯 가지가 들어 있다. 시스템 프롬프트, 도구 정의, 검색된 문서, 대화 기록, 도구 출력이다.

| 구성요소 | 역할 | 문제 |
| --- | --- | --- |
| 시스템 프롬프트 | 모델의 성격과 규칙 정의 | 건드리면 기본 행동이 바뀜 |
| 도구 정의 | 사용 가능한 도구 목록 | MCP가 많으면 수천 토큰 점유 |
| 검색된 문서 | RAG/JIT로 가져온 지식 | 관련 없는 것도 섞임 |
| 대화 기록 | 이전 턴들의 누적 | 길어지면 Lost-in-Middle 발생 |
| 도구 출력 | 파일 읽기, 검색 결과 등 | **전체의 80%+ 차지** |

마지막 항목이 결정적이다. 도구 출력이 전체의 80%+를 차지한다. AI에게 파일 10개를 읽게 시키면 그 10개 파일의 전체 내용이 컨텍스트에 들어간다. 파일 하나가 1,000줄이면 10,000줄, 토큰으로 수만 토큰이다. 실제로 필요한 부분은 100줄 정도일 것이다. 나머지 9,900줄은 노이즈다. 이 노이즈가 쌓이면 Chroma가 발견한 불균일한 성능 붕괴가 발생한다.

이것이 프롬프트 엔지니어링으로 해결 불가능한 이유다. 프롬프트가 아무리 좋아도, 컨텍스트 윈도우에 5만 토큰 이상의 무관한 도구 출력이 쌓여 있으면 모델의 행동이 예측 불가능해진다. 문제는 프롬프트가 아니라 프롬프트를 둘러싼 나머지 99.75%에 있다.

## 4축 프레임워크: Write, Select, Compress, Isolate

Lance Martin이 2025년 6월 23일에 제시한 4축이 컨텍스트 엔지니어링의 표준 프레임워크가 됐다. LLM을 운영체제에, 윈도우를 RAM에 비유한 분류다.

> **핵심**: 4축은 *Write*(윈도우 밖에 저장), *Select*(필요한 조각만 꺼내옴), *Compress*(토큰 축소), *Isolate*(컨텍스트 분리)다. 이 네 축을 조합해 유한한 윈도우 안에서 최대 신호 대 잡음비를 달성한다.

*Write 축*은 컨텍스트를 윈도우 밖에 저장하는 것이다. 컨텍스트 윈도우는 유한하므로 필요 없는 것은 밖으로 빼야 한다. Claude Code의 CLAUDE.md가 대표적 구현이다. 프로젝트 루트에 규칙 파일을 저장하면 매 세션 시작 시 자동으로 로드된다. 윈도우를 상시 점유하지 않으면서도 필요할 때 항상 참조할 수 있다. 2025년 9월 Anthropic이 이 원칙을 플랫폼에 적용한 Context Editing과 Memory Tool은 장기 에이전트 성능을 39% 향상시켰다.

*Select 축*은 필요한 조각만 꺼내오는 것이다. 프로젝트의 모든 파일을 읽는 대신, grep으로 특정 함수만 추출해 100줄만 로드한다. 도구 설명 RAG라는 기법도 있다. MCP로 도구가 30개 노출됐을 때 현재 작업에 관련된 도구만 검색해서 노출하면 도구 선택 정확도가 3배 올라간다.

*Compress 축*은 같은 의미를 더 적은 토큰으로 전달하는 것이다. Claude Code는 컨텍스트가 95% 임계값을 넘으면 자동으로 이전 대화를 요약한다. 관찰 마스킹(Observation Masking)이라는 기법은 도구 출력을 사용한 후 상세 내용을 압축 참조로 대체한다. 1,200줄짜리 파일 내용이 한 줄 요약으로 바뀌면 60~80% 토큰 절감이 가능하다. 품질 손실은 5% 미만이다.

*Isolate 축*은 하나의 거대한 컨텍스트 대신 여러 개의 격리된 컨텍스트로 나누는 것이다. 메인 에이전트가 전체를 관리하고, 서브에이전트가 각자 독립된 컨텍스트에서 탐색 작업을 수행한 뒤 1,000~2,000 토큰의 응축 요약만 메인으로 돌려보낸다.

여기서 중요한 논쟁이 있다. Cognition(Devin 팀)은 멀티에이전트를 쓰지 말라고 권고하고, Anthropic은 병렬 서브에이전트로 리서치 성능이 90% 향상됐다고 보고한다. 이 둘은 모순이 아니다. Cognition이 말하는 것은 컨텍스트를 *공유하려는* 멀티에이전트의 실패이고, Anthropic이 말하는 것은 컨텍스트를 *격리하는* 서브에이전트의 성공이다. 분리해야 하는 것은 역할이 아니라 컨텍스트다. *코드 작성자*와 *리뷰어*로 역할을 나누면 실패한다. 둘이 같은 컨텍스트를 공유해야 하기 때문이다. *메인 작업*과 *독립 탐색 작업*으로 컨텍스트를 나누면 성공한다.

## 압축의 우선순위: 뭘 먼저 줄일 것인가

4축 중 Compress를 쓸 때 무엇을 먼저 압축해야 하는지 순서가 있다. 도구 출력이 80%+를 차지하므로 가장 먼저 압축한다. 두 번째는 오래된 턴이다. 세 번째는 검색된 문서다. 시스템 프롬프트는 절대 압축하지 않는다. 시스템 프롬프트를 압축하면 모델의 기본 행동이 바뀌기 때문이다.

도구 출력에서 절감하는 효과가 가장 크다. 파일을 읽은 후에 핵심만 남기고 나머지를 요약하면 60~80% 절감이 가능하다. KVTC(2025)라는 기법은 신호 처리 기법으로 KV-Cache를 20~40배 손실 압축한다. 프레임워크 수준에서 작동하는 기법이라 사용자가 직접 쓸 일은 없지만, 앞으로 컨텍스트 비용이 구조적으로 낮아질 것이라는 신호다.

## JIT 컨텍스트: RAG의 진화

전통적 RAG는 쿼리 직전에 임베딩 검색으로 문서를 미리 모아 프롬프트에 붙이는 방식이다. JIT(Just-in-Time) 컨텍스트는 다르다. 에이전트가 파일 경로나 쿼리 같은 식별자만 들고 있다가, 필요할 때 도구로 런타임에 적재한다. Claude Code의 head, tail, grep, glob, CLAUDE.md 하이브리드가 이 방식의 실전 구현이다.

*긴 컨텍스트 LLM이면 RAG가 필요 없다*는 이분법이 한때 유행했다. Li 등의 연구가 이것을 반증했다. 자원이 충분하면 긴 컨텍스트(LC)가 평균 성능에서 RAG를 앞서지만, RAG의 비용 우위는 그대로다. 대화형 질의와 일반 질의에서는 RAG가 여전히 유리하다. 또한 단순 청킹(naive chunking) RAG는 LC에 지지만, 요약 기반 검색을 적용하면 LC와 비등해진다. 진짜 변수는 *LC냐 RAG냐*가 아니라 *검색 전략 선택*인 셈이다. 실무 최적은 모델이 쿼리를 동적으로 라우팅하는 Self-Route 하이브리드다.

## 실전 사례: CLAUDE.md가 모든 것을 바꾼다

추상적 원칙만 나열하면 와닿지 않는다. Claude Code를 쓰는 사람이라면 CLAUDE.md를 알 것이다. 프로젝트 루트에 넣으면 매 세션 시작 시 자동으로 컨텍스트에 로드된다.

나쁜 CLAUDE.md는 이렇다. *이 프로젝트는 Next.js를 사용한다, 테스트를 먼저 실행하라, TypeScript를 사용한다.* 이것은 프롬프트다. *잘 해줘*와 다를 바 없다. 좋은 CLAUDE.md는 다르다. 파일 경로는 절대 경로만 사용할 것, 테스트는 단일 파일 먼저 돌리고 통과 후 전체를 돌릴 것, 커밋 훅 실패 시 원인을 수정하고 재시도할 것, 금지 사항(console.log 디버깅, any 타입)을 명시한다. 아키텍처 정보도 넣는다. API는 어디에, DB는 Supabase이고 마이그레이션은 어디에, 인증은 JWT와 refresh token rotation이라는 것까지.

차이는 *뭘 쓰는지*가 아니라 *어떻게 작업해야 하는지*를 알려주는 데 있다. Anthropic이 SWE-bench에서 발견한 사실이 있다. 상대 경로를 절대 경로로 바꾸는 것만으로 도구 사용 오류가 완전히 사라졌다. 도구 하나의 규칙이 전체 작업 품질을 좌우한 셈이다. CLAUDE.md는 Write 축의 구현이다. 규칙을 파일(윈도우 밖)에 저장하되, 매 세션 자동 로드로 항상 어텐션이 높은 위치에 배치하는 것이다.

## 실전: 100개 파일 리팩토링에 4축을 적용하면

추상적 원칙을 실전에 대입해보자. Next.js 앱에서 API 라우트 100개를 새 인증 시스템으로 마이그레이션해야 하는 상황이다. 프롬프트만으로 시도하면 AI가 파일 5개쯤 수정하다가 컨텍스트가 터진다. 100개 파일의 내용을 동시에 들고 있을 수 없기 때문이다.

| 축 | 적용 | 효과 |
| --- | --- | --- |
| Write | CLAUDE.md에 마이그레이션 규칙 저장. 기존 패턴과 새 패턴, 완료 기록 방식을 명시 | 매 세션마다 자동 로드. 100번 반복 설명 불필요 |
| Select | 파일 100개를 한번에 읽지 않고 grep으로 대상만 추출, 3~5개씩 로드 | 필요한 것만 로드해 윈도우 폭발 방지 |
| Compress | 파일 수정 완료 후 도구 출력을 한 줄 요약으로 대체 | 1,200줄 원본 대신 한 줄. 60~80% 절감 |
| Isolate | 메인 에이전트가 계획을 관리하고, 서브에이전트가 폴더별로 독립 작업 | 각자 깨끗한 컨텍스트에서 작업 후 결과만 반환 |

이렇게 하면 컨텍스트가 터지지 않고 100개 파일을 처리할 수 있다. 프롬프트를 잘 쓰는 것만으로는 절대 불가능한 작업이다. 컨텍스트 엔지니어링이란 *정보를 언제, 어떤 형태로, 얼마나, 어디에 배치할 것인가*를 설계하는 행위다.

## 70~80%의 법칙과 KV-Cache

컨텍스트 윈도우를 얼마나 채워야 하는가. 답은 70~80%까지다. 그 이상 채우면 성능이 무너지기 시작한다. Claude Code는 95%에서 자동 압축을 발동하지만 이것은 비상 브레이크다. 정상 운전에서는 70~80%를 넘기지 않아야 한다.

Claude Opus 4.6과 Sonnet 4.6은 1M 토큰 윈도우를 갖고 있다. 큰 윈도우니까 다 쓸 수 있을 것 같지만, Chroma 연구를 기억해야 한다. 200K 모델이 50K에서 이미 무너졌다. 1M 모델이라고 1M까지 안전한 것은 아니다.

이 제한이 존재하는 이유 중 하나가 KV-Cache 비용이다. 트랜스포머는 매 토큰 생성 시 이전 토큰들의 Key-Value 벡터를 참조한다. 이것이 KV-Cache다. 컨텍스트가 길수록 캐시가 커지고, 비용이 비선형으로 증가한다. 10K 토큰에서 100K 토큰으로 늘리면 비용이 10배가 아니라 그 이상 올라간다.

이 비용을 줄이는 핵심 기법이 프리픽스 캐싱(Prefix Caching)이다. 해시 기반 블록 매칭으로 동일한 프리픽스를 재사용하는 것이다. 시스템 프롬프트와 도구 정의처럼 매 요청마다 반복되는 부분을 캐싱하면 비용이 대폭 줄어든다. 이것이 컨텍스트 배치 순서에 영향을 준다. 안정적인 것을 앞에, 변하는 것을 뒤에 배치하면 캐시 히트율이 올라가고 비용이 내려간다. CLAUDE.md가 맨 앞에 로드되는 이유, 시스템 프롬프트가 항상 컨텍스트의 맨 앞에 있는 이유가 여기에 있다.

> **핵심**: 컨텍스트 윈도우는 70~80%까지만 채운다. 안정적인 요소를 앞에, 변하는 요소를 뒤에 배치해 프리픽스 캐싱 효율을 극대화한다. 비용과 성능을 함께 고려하면 *필요한 최소한*만 넣는 것이 항상 최선이다.

## 컨텍스트 엔지니어링은 건축이다

왜 *엔지니어링*이라는 이름이 붙었는지를 짚어야 한다. 프롬프트 엔지니어링에서의 *엔지니어링*은 솔직히 과장이었다. 문장을 잘 쓰는 것은 작문이지 엔지니어링이 아니다. 하지만 컨텍스트 엔지니어링은 진짜 엔지니어링이다.

Anthropic이 제시한 핵심 원리가 있다. 좋은 컨텍스트란 *원하는 결과의 가능성을 최대화하는 가장 작은 고신호 토큰 집합*이다. 이것은 정적 템플릿이 아니라 개발자, 사용자, 이전 상호작용, 도구 호출, 외부 데이터가 런타임에 조립되는 파이프라인이다. Harrison Chase의 정의를 빌리면, *LLM이 태스크를 그럴듯하게 해낼 수 있도록 올바른 정보와 도구를 올바른 형식으로 제공하는 동적 시스템을 구축하는 것*이다.

프롬프트를 쓰는 것이 페인트칠이라면, 컨텍스트를 설계하는 것은 건물의 골조를 세우는 것이다. 페인트칠도 중요하다. 하지만 골조가 부실하면 건물이 무너진다.

## 컨텍스트 엔지니어링의 한계, 그리고 다음

컨텍스트 엔지니어링은 프롬프트 엔지니어링의 한계를 넘어선 전환이다. 하지만 이것도 한계가 있다. 수동이라는 한계다.

Write, Select, Compress, Isolate 4축을 사람이 일일이 관리하는 것은 단발성 대화에서는 가능하다. *이 파일을 읽어서 이렇게 요약해줘*라고 말하는 것은 Select와 Compress를 수동으로 지시한 것이다. 그러나 100개 파일 리팩토링의 3시간짜리 세션에서, 일주일짜리 프로젝트에서 사람이 매 스텝마다 *지금 뭘 로드하고 뭘 버려*를 지시하는 것은 불가능하다.

이것을 자동화하는 소프트웨어가 필요하다. 컨텍스트를 모니터링하고, 예산을 관리하고, 필요 시 압축하고, 서브에이전트로 격리하는 소프트웨어 껍데기. 그것이 하네스(Harness)다. 다음 장에서 다룬다.
