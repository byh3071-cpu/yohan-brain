# 하네스: AI가 일하게 만드는 소프트웨어

*앞 장에서 컨텍스트를 수동으로 관리하는 것의 한계를 확인했다. 자동화가 필요하다. 그 자동화를 담당하는 소프트웨어가 하네스다. 같은 모델이라도 하네스의 유무에 따라 채팅봇이 되기도 하고, 자율 에이전트가 되기도 한다.*

## 세계에서 가장 똑똑한 모델이 가장 무능해지는 순간

Claude.ai 채팅 인터페이스에서 *이 프로젝트의 auth 모듈을 리팩토링해줘*라고 말해보자. 돌아오는 답은 *프로젝트의 코드를 보여주시면 도와드리겠습니다*다. 파일을 읽을 수 없고, 코드를 실행할 수 없고, 결과를 검증할 수 없다. 세계에서 가장 똑똑한 모델이 아무것도 못 하는 상태다.

같은 말을 Claude Code에서 해보자. Claude Code는 프로젝트 구조를 탐색하고(Glob, Grep), auth 관련 파일을 읽고(Read), 코드를 수정하고(Edit), 테스트를 실행하고(Bash), 실패하면 수정하고 다시 테스트한다. 같은 모델이다. Claude Opus 4.6. 달라진 것은 모델을 감싸는 소프트웨어, 즉 하네스다.

Boris Cherny(Anthropic, Claude Code 초기 개발자)가 이것을 *Product Overhang*이라 불렀다. *모델의 능력은 이미 충분한데, 그 능력을 제품으로 만드는 소프트웨어가 없었다.* Cherny가 처음 Claude에게 파일시스템 접근 권한을 줬을 때 *Claude가 파일시스템을 탐색하는 것이 경이로웠다*고 회고한 바 있다. 파일을 읽게 해줬더니 모든 것이 달라진 것이다.

> **핵심**: 하네스(Harness)란 LLM을 보이지 않는 프롬프트와 호출 가능한 도구로 확장하는 소프트웨어다(Simon Willison, 2026). 모델은 같은데 제품이 바뀐다. 채팅봇과 에이전트의 차이는 모델의 차이가 아니라 하네스의 차이다.

## 하네스는 두 가지를 한다

Simon Willison이 2026년 3월 16일에 정의한 것처럼, 하네스의 기능은 두 가지로 압축된다.

첫째, 보이지 않는 프롬프트를 관리한다. 사용자가 Claude Code에서 *이 파일 고쳐줘*라고 한 줄을 치면, 실제로 모델에게 전달되는 것은 그 한 마디가 아니다. 시스템 프롬프트 약 5,000 토큰, CLAUDE.md 내용, 도구 정의 목록, 이전 대화 기록이 앞에 붙는다. 사용자는 한 줄을 썼지만 모델은 수만 토큰을 받은 셈이다. 이 보이지 않는 부분을 관리하는 것이 하네스의 첫 번째 역할이다.

둘째, 호출 가능한 도구를 제공한다. 모델이 외부 세계와 상호작용하는 통로다. 모델이 *auth.ts 파일을 읽어야겠다*고 판단하면 Read 도구를 호출하고, 하네스가 실제로 파일을 읽어서 결과를 컨텍스트에 넣는다. 수정이 필요하면 Edit 도구를 호출하고, 테스트가 필요하면 Bash 도구를 호출한다. 모델이 *이 도구를 이 인자로 호출해줘*라고 요청하고, 하네스가 실제로 실행하고 결과를 돌려준다.

LLM은 무상태(stateless)라는 점이 핵심이다. Willison이 지적한 것처럼, *LLM은 매번 실행될 때마다 같은 백지에서 시작한다.* 대화가 이어지는 것처럼 보이지만 실제로는 하네스가 매번 전체 대화 이력을 다시 전송한다. 토큰 캐싱이 비용을 줄여주긴 하지만, 구조적으로 모든 턴마다 전체 컨텍스트가 재구성된다. 이 재구성을 자동으로 처리하는 것이 하네스의 일이다.

이 두 가지가 합쳐지면 채팅봇이 에이전트가 된다.

## CLAUDE.md: 프로젝트의 두뇌를 설계하는 방법

보이지 않는 프롬프트 중 가장 중요한 것이 CLAUDE.md다. 프로젝트 루트에 놓는 파일 하나가 에이전트의 행동 전체를 바꾼다. CLAUDE.md는 하네스가 매 턴마다 자동으로 읽어서 시스템 프롬프트에 주입하는 프로젝트별 설정 파일이다.

나쁜 CLAUDE.md는 이렇게 생겼다.

| [*나쁜 예: 추상적이고 실행 불가능*],  [코드를 잘 짜주세요. 테스트도 꼼꼼히 하고 좋은 코드를 작성하세요. 사용자 경험을 중시합니다.],)

이것은 아무 정보도 주지 않는다. *좋은 코드*가 무엇인지, *테스트를 꼼꼼히*가 어떤 기준인지 모델은 알 수 없다. 주니어 개발자에게 이렇게 말하면 좋은 결과가 나오지 않는 것과 같다.

좋은 CLAUDE.md는 이렇게 생겼다.

| [*좋은 예: 구체적이고 실행 가능*],  [*빌드/테스트 명령어*: npm run build, npm test, 단일 테스트는 npm test -- --grep "test name". *코드 스타일*: TypeScript strict mode, 함수형 우선, any 금지. 에러는 Result 타입으로 처리. *커밋 규칙*: conventional commits (feat:, fix:, refactor:). 커밋 전 반드시 npm test 통과 확인. *프로젝트 구조*: src/api/ (라우트), src/domain/ (비즈니스 로직), src/infra/ (DB, 외부 서비스). domain은 infra를 직접 import하지 않는다.],)

좋은 CLAUDE.md에는 네 가지가 들어간다. 빌드와 테스트 실행 방법, 코드 스타일과 규칙, 프로젝트의 디렉토리 구조와 의존성 방향, 그리고 커밋 규칙이다. 모델이 첫 턴에서 *이 프로젝트는 어떻게 돌아가는가*를 즉시 파악할 수 있어야 한다. CLAUDE.md가 잘 쓰여 있으면 모델이 프로젝트 탐색에 쓰는 턴 수가 줄어들고, 그만큼 실제 작업에 더 많은 컨텍스트 예산을 쓸 수 있다.

Simon Willison의 관련 개념인 AGENTS.md도 같은 원리다. 에이전트가 사용할 수 있는 패키지, 환경 정보, 예시 명령어를 문서화한 것이다. 핵심은 동일하다. 에이전트가 시행착오로 알아내야 할 것을 문서로 미리 알려주면 루프가 빨라진다.

## 도구 정의 설계: ACI가 프롬프트보다 중요하다

에이전트에게 도구를 주는 것만으로는 부족하다. 도구를 *어떻게 정의하느냐*가 결과를 결정한다.

Anthropic의 Erik Schluntz와 Barry Zhang(2024)이 SWE-bench를 풀면서 발견한 것이 있다. *프롬프트보다 도구 최적화에 더 많은 시간을 썼다.* 구체적으로, 파일 편집 도구에서 상대 경로를 절대 경로로 바꾸는 것만으로 도구 사용 오류가 완전히 사라졌다. 에이전트가 디렉토리를 이동한 후에 *auth.ts*를 수정하면 잘못된 위치의 파일을 수정하거나 파일을 아예 못 찾는다. /home/user/project/src/auth/auth.ts라고 절대 경로로 명시하면 이 문제가 사라진다. 사소해 보이는 변경이 전체 작업의 성패를 갈랐다.

Anthropic은 이 원칙을 ACI(Agent-Computer Interface) 디자인이라 부른다. 인간-컴퓨터 인터페이스(HCI)에 들이는 노력만큼을 에이전트-컴퓨터 인터페이스에도 들여야 한다는 것이다. 핵심 원칙은 세 가지다.

| [*원칙*], [*실천 방법*],  [모델의 입장에서 생각할 것], [모델이 도구를 처음 보는 주니어 개발자라고 상상하라. 암묵적 규칙은 없다],
  [주니어 개발자에게 설명하듯 문서를 쓸 것], [파라미터 이름, 반환값 형식, 에러 시 행동을 명시하라. 도구 설명은 수백 줄이 될 수 있다],
  [실수가 어렵도록 설계할 것(포카요케)], [상대 경로 대신 절대 경로를 강제하라. 위험한 기본값을 제거하라],)

도구 정의 자체가 컨텍스트 윈도우의 예산을 잡아먹는다는 점도 중요하다. Claude Code의 시스템 프롬프트에는 도구 정의만으로 수천 토큰이 들어간다. 도구가 많을수록 컨텍스트가 줄어든다. 도구는 적게, 하지만 정확하게 설계하는 것이 원칙이다.

Claude 4에서 관찰된 현상이 이 원칙을 뒷받침한다. 멀티에이전트 시스템에서 Claude 4가 자기 도구 설명을 자율적으로 재작성한 결과, 태스크 완료 시간이 40% 감소했다. 도구 설명이 곧 성능이라는 증거다. 나쁜 도구 설명은 나쁜 프롬프트보다 더 치명적이다. 프롬프트는 한 번 읽히지만, 도구 정의는 매 턴마다 컨텍스트에 포함되기 때문이다.

## 에이전트 루프: 수집, 행동, 검증, 반복

하네스의 심장은 루프다. 이 루프의 원형은 Yao 등(2022)의 ReAct 패턴이다. *추론 흔적(reasoning trace)*과 *행동(action)*을 교차시키는 것이다. 모델이 *무엇을 해야 하는지* 추론하고, 그에 따라 행동하고, 결과를 관찰하고, 다시 추론한다. 이 패턴이 현대 에이전트 루프의 기반이 됐다.

Anthropic의 Thariq Shihipar(2025)가 이 원형을 실전에 맞게 재정리한 패턴이 표준이 됐다. *Gather context, Take action, Verify work, Repeat.*

![워크플로우와 에이전트의 차이. 워크플로우는 경로가 사전에 고정돼 있고, 에이전트는 모델이 상황에 따라 다음 행동을 스스로 결정한다.](/assets/consulting-cropped/10-workflow-agent.png)

실전 시나리오로 이 루프를 따라가보자. *로그인이 안 된다는 버그 리포트*가 들어왔다.

| [*단계*], [*행동*], [*내용*],  [Gather], [grep, Read], [src/ 에서 login 키워드 검색. 관련 파일 5개 발견. login.ts 코드 확인],
  [Think], [추론], [42번 줄에서 토큰 만료 체크가 빠져 있음을 발견],
  [Act], [Edit], [login.ts 42번 줄 수정],
  [Verify], [Bash], [npm test auth.test.ts 실행. 1건 실패],
  [Think], [추론], [타임존 문제 파악. UTC 변환 필요],
  [Act], [Edit], [login.ts 재수정],
  [Verify], [Bash], [auth 테스트 통과. 전체 테스트 통과],)

이 전체 과정에서 사람이 개입한 것은 처음 *로그인 버그 고쳐줘* 한 마디뿐이다. 나머지는 하네스의 에이전트 루프가 자동으로 돌린 것이다. 수집(grep, Read), 행동(Edit), 검증(Bash: test), 실패 시 다시 수집부터.

검증 단계가 빠지면 치명적이다. 검증 없는 에이전트는 *고쳤다*고 보고하면서 실제로는 망가뜨린 상태가 된다. Anthropic이 제시한 검증 3단계가 있다.

| [*단계*], [*방법*], [*특성*],  [1단계], [규칙 기반 (lint, type check, test)], [가장 빠르고 저렴. 자동화 용이],
  [2단계], [시각적 피드백 (브라우저 스크린샷)], [사람이 보는 것과 같은 확인. 중간 비용],
  [3단계], [LLM-as-Judge], [가장 비싸지만 유연. 복잡한 판단 가능],)

장기 작업에서는 단위 테스트보다 브라우저 기반 E2E 검증이 효과적이다. Justin Young(Anthropic, 2025)의 실험에서 Puppeteer MCP로 브라우저 자동화 도구를 제공하자 에이전트의 작업 품질이 크게 향상됐다. Playwright 스크린샷으로 실제 화면을 캡처해서 모델에게 *이게 맞는 결과인가*를 판단시키는 방식이다. 비용이 들지만, 인간이 보는 것과 가장 가까운 검증이다.

## 6.7B가 175B를 이기는 이유: 도구의 힘

도구 사용이 왜 중요한지를 보여주는 결정적 연구가 있다. Schick 등(Meta AI, 2023)의 Toolformer다. 자기지도학습으로 도구 사용법을 배운 모델이 후보 도구 호출을 생성하고 실행한 뒤, 미래 토큰 퍼플렉시티가 낮아지는 것만 보존하는 방식으로 학습했다. 결과는 놀라웠다. 6.7B 파라미터에 도구를 결합해 GPT-3 175B 수준의 성능을 달성했다. 수학, 질의응답, 시간 추론에서 모델 크기의 25배 차이를 도구 사용이 메운 셈이다.

비유하면 이렇다. 암산만으로 복잡한 수학 문제를 풀려는 천재와, 계산기를 든 보통 사람이 경쟁하는 상황이다. 계산기가 있으면 보통 사람도 천재를 이길 수 있다. LLM에게 도구는 계산기와 같다. 모델의 내재 능력을 키우는 것보다 적절한 도구를 주는 것이 때로 더 효과적이다.

## 하네스 설정 과정: 새 프로젝트에서 시작하기

새 프로젝트에서 하네스를 세팅하는 과정은 네 단계로 나뉜다. 이 과정을 제대로 밟으면 에이전트가 처음부터 맥락을 가진 채 작업을 시작한다. 대충 밟으면 에이전트가 매번 프로젝트를 탐색하는 데 컨텍스트의 절반을 낭비한다.

*1단계: CLAUDE.md 작성.* 프로젝트 루트에 CLAUDE.md를 만든다. 빌드 명령어, 테스트 명령어, 코드 스타일 규칙, 디렉토리 구조를 적는다. 이것이 에이전트의 첫 번째 지식이 된다. 기존 프로젝트라면 README에서 핵심 정보를 추출해 에이전트용으로 재구성하면 된다. 핵심은 *사람이 읽는 문서*가 아니라 *에이전트가 실행하는 지침*이라는 점이다.

*2단계: 도구 범위 결정.* 에이전트가 사용할 도구의 범위를 정한다. 파일 읽기와 쓰기는 기본이다. 셸 명령 실행을 허용할 것인가, 브라우저 자동화를 붙일 것인가, 외부 API 호출을 허용할 것인가를 결정한다. Simon Willison의 조언대로, MCP에 의존하기보다 셸 명령으로 해결할 수 있는 것은 셸 명령으로 하는 편이 투명하고 디버깅이 쉽다.

*3단계: 권한 경계 설정.* 읽기 작업은 자동 허용, 쓰기 작업은 상황에 따라, 파괴적 작업은 반드시 승인으로 나눈다. YOLO 모드를 쓴다면 반드시 샌드박스 환경이나 스테이징 크리덴셜과 짝지어야 한다. Willison은 Fly.io 실험에서 에이전트 전용 조직을 5달러 예산 한도로 만들어 격리한 사례를 소개했다.

*4단계: 검증 루프 구축.* 테스트 명령어가 있으면 에이전트가 매 수정 후 자동으로 검증할 수 있다. 테스트가 없는 프로젝트라면 lint와 type check라도 설정한다. 검증 도구가 없으면 에이전트 루프의 핵심이 빠진 것이다. 자동화된 테스트가 에이전트의 가치를 극적으로 증폭한다는 것이 Willison의 핵심 관찰이다.

이 네 단계를 한 마디로 요약하면 이렇다. 에이전트가 *무엇을 알아야 하는지*(CLAUDE.md), *무엇을 할 수 있는지*(도구), *무엇을 하면 안 되는지*(권한), *어떻게 확인하는지*(검증)를 미리 정해주는 것이다.

## Before/After: 하네스의 유무가 만드는 차이

같은 모델, 같은 작업을 하네스 없이 했을 때와 있을 때를 비교하면 차이가 극명하다.

| [*작업*], [*하네스 없이 (채팅)*], [*하네스 있게 (에이전트)*],  [버그 수정], [사용자가 코드를 복사해 붙여넣고, 모델의 수정안을 다시 복사해 파일에 붙여넣고, 직접 테스트를 돌린다. 실패하면 에러 로그를 다시 복사해 대화에 붙여넣는다], [사용자가 *로그인 버그 고쳐줘* 한 마디. 에이전트가 파일 탐색, 수정, 테스트, 재수정을 자동으로 반복한다],
  [리팩토링], [파일마다 하나씩 보여주고, 수정안을 받아 적용하고, 다음 파일을 보여준다. 10개 파일이면 20번의 복사-붙여넣기], [에이전트가 프로젝트 전체를 탐색해 관련 파일을 찾고, 의존성을 분석해 순서대로 수정하고, 전체 테스트를 돌려 확인한다],
  [새 기능 개발], [사용자가 요구사항을 설명하고, 모델이 코드를 생성하면 사용자가 파일에 넣고 실행해본다. 오류가 나면 다시 대화], [사용자가 요구사항을 설명하면 에이전트가 파일 생성, 라우트 추가, 테스트 작성, 실행, 수정까지 자동으로 진행한다],)

차이의 본질은 *누가 루프를 도는가*다. 하네스 없이는 사람이 수집-행동-검증 루프의 모든 단계를 직접 실행한다. 하네스가 있으면 소프트웨어가 루프를 돌고 사람은 시작 명령만 내린다. 같은 버그 수정에 채팅에서는 30분이 걸리고 에이전트에서는 3분이 걸리는 이유가 여기에 있다. 모델의 추론 속도가 빨라진 것이 아니다. 사람이 병목이던 구간이 자동화된 것이다.

## 장기 실행 에이전트: 하루짜리 작업을 맡기려면

5분짜리 버그 수정이 아니라 3시간짜리 기능 개발이나 하루짜리 리팩토링을 에이전트에게 맡기려면 한 단계 더 나가야 한다. LLM은 무상태(stateless)다. Justin Young(Anthropic, 2025)의 표현대로, *새 세션은 이전에 무엇을 했는지 아무 기억 없이 시작된다.* 하지만 git commit과 progress 파일은 유상태(stateful)다. 세션이 끊겨도 살아 있다.

Young이 이 문제를 2부 구조로 해결했다.

*초기화 에이전트(Initializer).* 프로젝트를 처음 받으면 init.sh 스크립트를 실행하고, claude-progress.txt 파일을 생성하고, 200개 이상의 기능 요구사항을 JSON으로 정리하고, 초기 git commit을 만든다. 모든 기능은 처음에 *실패* 상태로 표시된다. 이것이 에이전트에게 명확한 작업 목록과 진척 측정 기준을 제공한다.

*코딩 에이전트(Coder).* 매 세션 시작 시 다음 순서를 따른다. 현재 디렉토리를 확인하고, git log를 리뷰해 이전 세션에서 무엇을 했는지 파악하고, claude-progress.txt를 읽어 미완료 기능을 찾고, 개발 서버를 띄운 뒤 기존 기능이 깨지지 않았는지 먼저 확인하고, 그 다음에 새 기능 하나를 구현한다.

이 구조의 핵심은 *파일시스템이 메모리*라는 점이다. LLM의 컨텍스트 윈도우는 세션이 끝나면 사라지지만, git history와 progress 파일은 영속한다. 하네스가 무상태 LLM의 한계를 유상태 외부 시스템으로 극복하는 셈이다.

장기 실행에서 발견된 네 가지 실패 모드와 대응이 있다.

| [*실패 모드*], [*증상*], [*대응*],  [너무 많은 동시 작업], [구현 중 컨텍스트 소진. 절반만 완성된 기능이 여러 개], [JSON 기능 목록으로 단일 기능 단위 작업. 한 번에 하나만],
  [조기 완료 선언], [절반도 안 됐는데 *완료*라고 보고], [브라우저 자동화로 자가 검증. 화면에서 직접 확인],
  [문서 없는 코드 방치], [나중에 아무도 이해 불가], [구현 완료 시 README와 progress 파일 갱신 의무화],
  [앱 실행 방법 미숙지], [에이전트가 서버를 못 띄움], [init 스크립트에 시작 절차 명시. CLAUDE.md에 실행 명령어 기록],)

두 번째 실패 모드가 특히 치명적이다. 에이전트가 테스트를 지우거나 수정해서 *통과*시키는 경우, 아무것도 검증되지 않은 것이다. Young의 명시적 경고가 있다. *에이전트가 테스트를 지우거나 수정하게 허용하는 것은 용납할 수 없다.* 검증 루프의 기반 자체가 사라지기 때문이다.

## 서브에이전트: 컨텍스트를 분리하는 기술

하나의 에이전트가 모든 것을 하면 컨텍스트가 빠르게 소진된다. 10개 파일을 읽고, 3개를 수정하고, 테스트를 돌리면 컨텍스트 윈도우의 상당 부분이 이미 찬 상태다. 이때 추가 작업을 맡기면 앞서 읽은 파일의 내용이 컴팩션(요약)되면서 세부 정보가 사라진다.

서브에이전트는 이 문제의 해법이다. 메인 에이전트가 특정 작업을 격리된 컨텍스트 창에서 실행되는 서브에이전트에게 위임하는 패턴이다. Claude Code에서 이 기능은 3일 만에 구축됐고, 개발 과정에서 2일치 작업이 폐기됐다(Boris Cherny, 2025). 빠르게 시도하고 빠르게 버리는 개발 방식의 전형이다.

서브에이전트의 원리는 이렇다. 메인 에이전트가 *이 파일의 타입 에러를 모두 수정해*라는 작업을 서브에이전트에게 넘긴다. 서브에이전트는 자기만의 컨텍스트 윈도우에서 해당 파일을 읽고, 수정하고, 검증한다. 완료되면 결과 요약만 메인 에이전트에게 돌려준다. 메인 에이전트의 컨텍스트에는 서브에이전트가 읽은 파일 전체가 아니라 *타입 에러 5건 수정 완료*라는 요약만 들어간다.

이 패턴이 유효한 경우와 그렇지 않은 경우가 있다.

| [*서브에이전트가 유효한 경우*], [*서브에이전트가 불필요한 경우*],  [독립적인 파일이나 모듈 단위 작업], [파일 간 의존성이 높아 전체 맥락이 필요한 작업],
  [탐색적 작업 (여러 디렉토리에서 정보 수집)], [한 파일 안에서의 단순 수정],
  [병렬 처리가 가능한 반복 작업], [사용자와의 대화 맥락이 중요한 작업],)

Anthropic의 Building Effective Agents 문서에서 제시한 *Orchestrator-Workers* 패턴이 이 구조의 일반형이다. 중앙 LLM이 작업을 동적으로 분해하고, 워커 LLM에게 위임하고, 결과를 합성한다. 서브에이전트는 이 패턴의 코딩 에이전트 구현이다.

장기 실행 에이전트에서 서브에이전트의 가치는 더 커진다. 메인 에이전트가 3시간 동안 작업하면서 컨텍스트가 누적되면, 새 작업을 깨끗한 컨텍스트에서 시작하는 서브에이전트가 오히려 더 정확하다. 오래된 컨텍스트에 묻혀 있는 메인 에이전트보다 방금 시작한 서브에이전트가 단일 작업에서는 더 나은 결과를 내는 역설이 발생한다.

## Claude Code의 숫자들: 하네스가 만드는 차이

Claude Code가 하네스의 대표 구현이라고 했다. 숫자로 보면 이 하네스가 얼마나 강력한지 실감할 수 있다. The Pragmatic Engineer의 Gergely Orosz가 Boris Cherny, Sid Bidasaria(창립 엔지니어), Cat Wu(PM)를 인터뷰한 기사에서 공개된 수치다.

| [*지표*], [*수치*],  [연간 매출 런레이트], [5억 달러 이상 (2025년 5월 일반 출시 이후)],
  [성장률], [3개월 만에 사용량 10배],
  [내부 도입 1일차], [Anthropic 엔지니어의 20%],
  [내부 도입 5일차], [50%],
  [자체 코드 비율], [Claude Code의 약 90%가 Claude Code로 작성],
  [릴리즈 속도], [내부 60~100회/일, 외부 1회/일],
  [엔지니어당 PR], [하루 약 5개],)

설계 철학에서 나온 핵심 원칙이 있다. *비즈니스 로직을 최소화하고, 모델이 대부분의 일을 한다. 새 모델이 나올 때마다 코드를 지운다.* 이것이 의미하는 바는 명확하다. 좋은 하네스는 모델을 바꿔도 제품이 유지된다. 나쁜 하네스는 모델을 바꾸면 전체를 다시 써야 한다.

기술 스택도 이 철학을 반영한다. TypeScript, React with Ink(터미널 UI), Bun(빌드). 이 조합을 선택한 이유가 있다. Claude가 이미 인터넷에서 가장 많이 본 기술, 즉 *분포 위에 있는(on distribution)* 기술이기 때문이다. 모델이 잘 아는 언어로 하네스를 만들면 모델이 자기 하네스를 개선할 수 있다. Claude Code의 90%가 Claude Code 자체를 이용해 작성됐다는 사실이 이 원칙의 증거다.

## 반패턴: 이렇게 하면 망한다

세 가지 대표적 반패턴이 있다.

첫째, 워크플로우를 에이전트라고 부르는 것이다. 입력을 받아 요약하고, 번역하고, 포맷팅하는 세 단계가 if-else로 고정돼 있으면 이것은 워크플로우다. Anthropic의 정의가 명확하다. *워크플로우는 LLM과 도구가 사전 정의된 코드 경로로 오케스트레이션되는 시스템이고, 에이전트는 LLM이 자신의 프로세스와 도구 사용을 동적으로 지휘하는 시스템이다.* 워크플로우는 경로가 코드에 박혀 있고, 에이전트는 경로가 모델의 추론에 의해 런타임에 결정된다. 워크플로우가 나쁜 것이 아니다. 잘 정의된 작업에는 워크플로우가 예측 가능하고 일관성 있다. 문제는 워크플로우를 에이전트라고 부르면서 에이전트의 유연성을 기대하는 것이다.

둘째, 비즈니스 로직을 하네스에 직접 넣는 것이다. 파이썬 파일이면 Black을 돌리고 타입스크립트 파일이면 Prettier를 돌리는 분기를 하네스에 넣으면 새 언어가 추가될 때마다 코드를 수정해야 한다. 이런 규칙은 CLAUDE.md나 Skills에 넣어야 한다. 하네스는 인프라여야 한다.

셋째, 테스트를 에이전트가 수정하거나 삭제하게 허용하는 것이다. AI가 *테스트를 수정해서 통과시켰습니다*라고 하면 아무것도 검증되지 않은 것이다. 검증 루프의 기반이 무너지기 때문이다.

> **핵심**: 좋은 하네스의 리트머스 테스트는 세 가지다. 모델을 교체해도 제품이 유지되는가. 비즈니스 로직이 하네스가 아니라 설정에 있는가. 에이전트가 테스트를 수정하지 못하게 막고 있는가.

## 에이전틱 루프 설계: 새로운 핵심 스킬

Simon Willison은 *에이전틱 루프를 설계하는 능력*이 2025년에 새로 이름 붙은 스킬이라고 말했다. Claude Code가 2025년 2월에 출시된 이후 불과 1년 만에 정의된 직업적 역량이다. 이 스킬의 핵심은 도구와 루프를 어떻게 조합하느냐에 있다.

효과적인 에이전틱 루프의 조건은 명확하다. 성공 기준이 분명해야 한다. 시행착오가 수반될 가능성이 높아야 한다. 자동화된 검증이 가능해야 한다. Willison이 열거한 이상적 후보 작업이 있다.

| [*작업*], [*왜 에이전틱 루프에 적합한가*],  [실패하는 테스트 디버깅], [성공 기준(테스트 통과)이 자동으로 측정된다],
  [느린 SQL 쿼리 최적화], [실행 시간이 수치로 나온다. 개선 여부가 즉시 확인된다],
  [의존성 업그레이드], [빌드 성공/실패가 자동 판정된다],
  [컨테이너 크기 축소], [이미지 크기가 숫자로 나온다. 기능 유지는 테스트로 확인한다],)

공통점이 있다. 전부 *기계가 판정할 수 있는 성공 기준*이 있다. 반대로, *이 디자인이 예쁜가*나 *이 문장이 자연스러운가*는 에이전틱 루프에 적합하지 않다. 기계가 판정할 수 없으면 루프가 돌 수 없다.

프레임워크 선택에 대해 Anthropic은 명확한 입장을 갖고 있다. LLM API 직접 호출로 시작하라는 것이다. Claude Agent SDK, AWS Strands Agents SDK 같은 프레임워크가 편의를 주지만, 프레임워크가 프롬프트와 응답을 가리면 디버깅이 무너진다. 추상화 계층이 편의를 주지만, 문제가 생겼을 때 어디서 잘못됐는지 알 수 없게 만든다. 간단한 에이전트 루프는 몇 줄의 코드로 구현할 수 있다. Willison의 표현대로 *LLM + 시스템 프롬프트 + 도구를 루프에 넣으면* 기본 구현은 끝이다.

## 하네스 경쟁: Claude Code만 있는 것이 아니다

2026년 현재 주요 하네스들이 경쟁하고 있다.

| [*하네스*], [*접근 방식*], [*특징*],  [Claude Code], [최소 비즈니스 로직, 자율 에이전트], [Anthropic 직접 개발. 모델이 대부분 결정],
  [Cursor], [IDE 내장형], [코드 인덱싱 + AI 자동완성이 핵심],
  [GitHub Copilot], [기존 워크플로우에 AI 삽입], [VS Code/JetBrains 통합],
  [Windsurf (Codeium)], [IDE 자체가 하네스], [플로우 기반 작업 관리],
  [Aider], [오픈소스 CLI], [가볍고 다중 모델 지원],)

공통점이 있다. 전부 *LLM + 파일시스템 + 도구 호출 + 검증 루프*라는 같은 뼈대를 공유한다. 차이는 어디에 중점을 두느냐다. IDE 통합(Cursor, Copilot)이냐, 자율 에이전트(Claude Code)냐.

## 권한과 안전: 돌이킬 수 없는 동작을 어떻게 다루는가

에이전트에게 도구를 주면 능력이 생기지만, 위험도 함께 생긴다. 파일을 수정할 수 있다는 것은 파일을 잘못 수정할 수도 있다는 뜻이다. 코드를 실행할 수 있다는 것은 위험한 명령어를 실행할 수도 있다는 뜻이다.

Simon Willison이 코딩 에이전트의 위험을 세 가지로 정리했다. 첫째, 나쁜 셸 명령이 중요한 파일을 삭제하거나 손상시키는 것. 둘째, 유출 공격(exfiltration attack)으로 에이전트가 볼 수 있는 파일이나 데이터를 외부로 빼내는 것. 셋째, 에이전트의 머신을 DDoS 프록시나 해킹 도구로 악용하는 것이다.

균형점은 이렇다. 읽기 작업(grep, Read)은 자동 허용한다. 쓰기 작업(Edit)은 상황에 따라 자동 또는 승인한다. 파괴적 작업(삭제, push, 프로덕션 접근)은 반드시 승인한다. YOLO 모드(전부 자동 허용)를 쓸 때는 샌드박스 환경, 스테이징 크리덴셜, 예산 한도와 짝지어야 한다.

위험 완화 옵션을 Willison은 세 가지로 나눴다. 샌드박스 환경을 사용하는 것, 남의 컴퓨터(GitHub Codespaces 같은)를 사용하는 것, 또는 위험을 감수하되 악성 지시 소스를 피하는 것이다. 이 균형을 잘못 잡으면 두 가지 극단에 빠진다. 하나는 모든 걸 허용해서 사고가 나는 것이고, 다른 하나는 모든 걸 막아서 에이전트가 무력해지는 것이다. 매 명령에 승인을 요구하면 에이전트가 시행착오로 문제를 풀어가는 능력, 즉 에이전트의 핵심 가치가 사라진다.

## 하네스의 다음: 외부 세계와의 연결

하네스가 AI를 *일하게* 만들었다. 파일을 읽고, 코드를 수정하고, 테스트를 돌리는 것은 이제 된다. 하지만 외부 시스템과의 연결은 다른 문제다. GitHub에서 이슈를 가져오고, Slack에서 메시지를 보내고, DB에서 쿼리를 실행하려면 하네스에 외부 세계의 팔을 달아주는 표준이 필요하다.

그 표준이 MCP(Model Context Protocol)다. 하지만 이 프로토콜에는 약속만큼이나 문제도 있었다. PART IV에서 다룬다.
