# 에이전트: 2025년 이후 가장 남용된 단어

*2025년부터 모든 AI 제품이 스스로를 에이전트라고 부르기 시작했다. 챗봇도, API 래퍼도, if-else 분기도 전부 에이전트다. 이 글은 에이전트의 실체를 짚는다. 진짜 에이전트는 무엇이 다르고, 왜 이 구분이 중요한지를 다룬다.*

## 에이전트라는 단어가 의미를 잃었다

Simon Willison은 2025년에 트위터에서 211개의 서로 다른 에이전트 정의를 수집했다. Michael Wooldridge는 이미 1994년에 *에이전트란 무엇인가라는 질문 자체가 에이전트 연구 커뮤니티에 민망하다*고 기록한 바 있다. 30년이 지났는데 상황이 나아지기는커녕 악화됐다.

2025년 AI 시장에서 벌어진 일을 보자. 챗봇에 버튼 두 개를 붙이면 에이전트다. API를 감싸서 프롬프트를 넣으면 에이전트다. if-else로 분기하는 워크플로우도 에이전트라고 부른다. 마케팅에서는 이 단어가 매력이 있다. *자율*, *지능*, *자동*이라는 인상을 주기 때문이다. 그러나 단어가 모든 것을 가리키면 아무것도 가리키지 않게 된다.

2025년 9월, Willison이 드디어 *쓸 만한 합의가 생겼다*고 선언했다. 그가 받아들인 최소 정의는 단 한 문장이다.

> **핵심**: LLM 에이전트(LLM Agent)란 *목표를 달성하기 위해 루프 안에서 도구를 실행하는 시스템*이다. 세 단어가 핵심이다: 도구(tools), 루프(loop), 목표(goal). 도구 없이 텍스트만 생성하면 챗봇이고, 루프 없이 한 번만 실행하면 파이프라인이다. 목표가 없으면 방향 없는 반복일 뿐이다.

이 정의가 좁아 보일 수 있다. Lilian Weng(OpenAI 연구자)은 2023년에 계획(Planning), 메모리(Memory), 도구 사용(Tool Use)의 세 축으로 에이전트를 구조화했고, Andrew Ng는 2024년에 성찰, 도구 사용, 계획, 멀티에이전트의 네 가지 설계 패턴을 제시했다. 이 분류들이 서로 모순되지는 않는다. Willison의 정의가 필요충분 조건이라면, Weng과 Ng의 분류는 에이전트를 *잘* 만들기 위한 설계 원칙인 셈이다.

Willison은 한 가지를 명확히 거부했다. *인간을 대체하는 존재로서의 에이전트*라는 정의다. OpenAI가 세 제품에 걸쳐 이 의미를 혼용했지만, Willison은 책임(accountability)이 인간 고유 속성이라 의미 있는 업무 위임은 사람에게만 가능하다는 입장을 고수했다.

## 워크플로우와 에이전트의 경계선

2024년 12월, Anthropic의 Erik Schluntz와 Barry Zhang이 하나의 구분선을 그었다. 이 구분이 에이전트를 이해하는 열쇠다.

택배 분류 시설을 생각해보자. 컨베이어 벨트 위의 상자가 무게 센서를 지나고, 바코드 스캐너를 거치고, 목적지별 슬라이드로 내려간다. 경로가 미리 설계되어 있다. 어떤 상자가 들어와도 센서, 스캐너, 슬라이드 순서는 바뀌지 않는다. 이것이 *워크플로우*다. 개발자가 경로를 결정하고, LLM은 각 정거장에서 지시받은 작업만 수행한다.

반면 배달 기사가 주소를 받고 출발하는 상황을 떠올려보자. 첫 번째 경로가 공사 중이면 우회한다. 수령인이 부재하면 경비실에 맡길지, 재방문할지를 판단한다. 판단의 결과에 따라 다음 행동이 달라진다. 이것이 *에이전트*다. 모델이 상황에 따라 경로를 스스로 결정한다.

| [*항목*], [*워크플로우*], [*에이전트*],  [경로 결정자], [개발자], [모델],
  [실행 순서], [고정 (A→B→C)], [동적 (상황에 따라 변경)],
  [오류 대응], [사전 정의된 분기], [자체 판단으로 방향 전환],
  [적합한 과제], [반복적, 예측 가능한 작업], [탐색적, 맥락 의존적 작업],)

이 표에서 주목할 것은 *경로 결정자*다. 워크플로우에서 LLM은 실행자일 뿐이다. 요약하라고 하면 요약하고, 번역하라고 하면 번역한다. 순서를 바꾸거나 단계를 건너뛸 권한이 없다. 에이전트에서 LLM은 의사결정자다. 다음에 무엇을 할지를 스스로 고른다.

현재 시장에서 에이전트라고 불리는 제품의 상당수가 실은 워크플로우다. 이름만 에이전트일 뿐, 경로가 코드에 고정되어 있다면 기술 분류상 워크플로우에 해당한다.

## 에이전트 루프: 생각하고 행동하고 확인하고 반복한다

에이전트의 심장은 루프(loop)다. Gather, Act, Verify, Repeat의 네 박자로 구성된다.

의사가 환자를 진료하는 과정을 생각해보자. 증상을 듣고(Gather), 검사를 지시하고(Act), 결과를 확인하고(Verify), 결과에 따라 추가 검사를 하거나 처방을 내린다(Repeat). 이 과정을 환자가 매번 지시하지 않는다. 의사가 스스로 판단해서 다음 단계로 넘어간다. 에이전트도 같은 방식이다. 사용자가 목표만 주면 에이전트가 루프를 자율 운행한다.

아래 그림은 에이전트 루프의 구조를 보여준다. 왼쪽의 *목표 입력*에서 시작해 Gather, Act, Verify를 거치며 순환하고, 종료 조건이 충족되면 빠져나온다.

![에이전트 루프의 구조. Gather → Act → Verify → Repeat 순환이 종료 조건을 만날 때까지 반복된다.](/assets/consulting-cropped/08-agent-loop.png)

이 그림에서 빨간색 화살표로 표시된 *Verify → Gather* 되돌림이 에이전트의 본질이다. 결과가 기대와 다르면 방향을 바꾼다. 워크플로우에는 이 되돌림이 없다.

루프가 영원히 돌면 문제가 된다. 종료 조건이 반드시 있어야 한다. 목표 달성, 컨텍스트 소진(토큰 예산 초과), 최대 반복 횟수 도달, 사용자 수동 정지. 이 네 가지 중 하나라도 빠지면 에이전트는 무한 루프에 빠진다. 실전에서 흔히 발생하는 사고가 바로 이것이다. AI가 같은 오류를 고치려고 무한히 시도하면서 API 비용만 치솟는 상황을 많은 개발자가 경험했다.

## 에이전트의 세 가지 능력: 도구, 메모리, 계획

에이전트를 에이전트답게 만드는 것은 세 가지 능력이다.

첫째, *도구 사용*이다. 파일 읽기, 코드 수정, 셸 실행, 웹 검색. Schick 등(Meta AI, 2023)의 Toolformer 연구가 이 능력의 위력을 보여준다. 6.7B 파라미터 모델에 도구를 붙이자 수학, 질의응답, 시간 추론 벤치마크에서 GPT-3(175B) 수준에 도달했다. 파라미터 수 차이가 25배인데, 도구가 그 격차를 메운 셈이다. 비행기에 비유하면 엔진 크기를 25배로 키우는 대신 날개를 달아 같은 고도에 오른 것과 같다.

둘째, *메모리*다. LLM은 본질상 무상태(stateless)다. 매 호출마다 백지에서 시작한다. 에이전트의 메모리는 외부 시스템에 의존한다. 작업 메모리는 현재 세션의 컨텍스트 윈도우다. 단기 메모리는 스크래치패드나 진행 파일이다. 장기 메모리는 CLAUDE.md나 Memory Tool처럼 세션을 넘어 유지되는 저장소다. 사람의 기억이 뇌에 있듯, 에이전트의 기억은 파일 시스템에 있다.

셋째, *계획*이다. *로그인 버그를 고쳐줘*라는 목표를 받으면 에이전트는 관련 파일 찾기, 원인 파악, 수정, 테스트의 순서를 스스로 세운다. 3단계에서 실패하면 계획을 수정한다. 워크플로우의 계획이 설계도면이라면, 에이전트의 계획은 전장의 작전이다. 설계도면은 바뀌지 않지만 작전은 전황에 따라 실시간으로 바뀐다.

Andrew Ng가 제시한 숫자가 이 세 능력의 효과를 압축한다. GPT-3.5의 HumanEval(코드 생성 벤치마크) zero-shot 성능은 48.1%다. GPT-4는 67.0%다. 그런데 GPT-3.5에 에이전트 루프를 붙이면 95.1%까지 올라간다. 모델 세대를 건너뛰는 것보다 루프와 도구를 붙이는 것이 더 큰 개선을 만든 셈이다.

## 멀티에이전트: 에이전트가 여럿이면 더 나은가

*에이전트 하나가 좋으면, 여럿이면 더 좋지 않은가.* 이 질문에 대한 답이 갈린다.

Cognition(Devin 팀)의 입장은 단호하다. *멀티에이전트를 쓰지 마라. 단일 스레드가 낫다.* 이유는 정보 손실이다. 에이전트 A가 발견한 정보를 에이전트 B에게 전달하는 과정에서 맥락이 유실된다. 컨텍스트 공유가 실패하면 두 에이전트가 같은 일을 중복 수행하거나 모순된 행동을 한다. 학교 조별과제에서 역할을 나눴는데 서로 소통이 안 되어 같은 부분을 두 번 작성하는 상황과 같다.

Anthropic의 입장은 다르다. 병렬 서브에이전트를 활용하면 리서치 성능이 90% 이상 개선된다. 단, 조건이 있다. *컨텍스트를 격리해야 한다.*

| [*패턴*], [*결과*],  [역할 기반 분리 (작성자/리뷰어)], [같은 컨텍스트를 공유해야 하므로 충돌 발생],
  [컨텍스트 기반 분리 (영역별 탐색)], [각자 독립 영역을 읽기 전용으로 탐색, 결과만 반환],)

이 표에서 성공하는 패턴의 공통점은 나누는 기준이 *역할*이 아니라 *컨텍스트*라는 점이다. 서브에이전트는 읽기 전용 탐색에 쓰고, 쓰기 작업은 메인 에이전트가 맡아야 한다. 오케스트라에 비유하면 각 파트가 자기 악보만 연주하고, 지휘자가 전체를 조율하는 구조다. 모든 연주자가 총보를 보면서 서로의 파트에 간섭하면 음악이 아니라 소음이 된다.

## 에이전트를 운영하면 생기는 문제

에이전트를 만드는 것은 시작이다. 운영이 진짜 도전이다.

비용 폭주가 첫 번째 문제다. 무한 루프에 빠지면 API 호출이 멈추지 않고, 청구서가 걷잡을 수 없이 불어난다. 데이터 유출이 두 번째다. 에이전트가 도구로 파일을 읽을 수 있다는 것은 민감한 정보에도 접근할 수 있다는 뜻이다. 의도치 않은 행동이 세 번째다. *파일 정리해줘*라는 요청이 중요한 파일 삭제로 이어질 수 있다. 관찰 불가능성이 네 번째다. 에이전트가 왜 그런 결정을 했는지 추적하기 어렵다.

이 네 가지를 다루는 분야를 AgentOps라고 부른다. 수돗물 공급 시스템에 비유하면 이해가 쉽다. 정수장을 짓는 것이 에이전트를 만드는 것이라면, 수질 검사, 유량 모니터링, 누수 탐지, 비상 차단 밸브를 설치하는 것이 AgentOps다. 토큰 모니터링, 행동 로그, 비용 한도, 위험 행동 알림이 여기에 해당한다.

Weng은 2023년에 에이전트의 세 가지 한계를 짚었다. 유한한 컨텍스트 윈도우가 역사와 지시를 전부 담지 못한다는 것, 장기 계획에서 사람만큼 견고하지 않다는 것, 자연어 인터페이스의 포맷 오류가 파싱 실패를 일으킨다는 것이다. 2026년 현재, 컨텍스트 윈도우가 100만 토큰 이상으로 확장됐지만 나머지 두 한계는 여전히 유효하다.

> **핵심**: 에이전트는 *목표, 루프, 도구*의 세 요소로 정의된다. 이 세 요소 중 하나라도 빠지면 에이전트가 아니다. 워크플로우와의 경계는 *경로를 누가 결정하는가*에 달려 있다. 코드가 결정하면 워크플로우이고, 모델이 결정하면 에이전트다.

다음 글에서는 이 에이전트를 실제로 만들 때 어떤 프레임워크를 선택해야 하는지를 다룬다. 선택지가 너무 많다는 것 자체가 이 분야의 미성숙함을 보여준다.
