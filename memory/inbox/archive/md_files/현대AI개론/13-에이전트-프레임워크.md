# 에이전트 프레임워크: 너무 많아서 선택할 수 없다

*에이전트를 만들겠다고 검색하면 이름이 쏟아진다. LangChain, LangGraph, CrewAI, AutoGen, PydanticAI, SmolAgents, Claude Agent SDK. 이 글은 각 프레임워크의 실체와 선택 기준을 정리한다. 결론부터 말하면, 프레임워크보다 중요한 것이 있다.*

## 프레임워크가 쏟아지는 이유

2024년에서 2026년 사이, 에이전트 프레임워크 생태계가 폭발했다. 위키 하나에 agent-sdk 관련 소스만 69개다. 이 숫자가 의미하는 바가 두 가지 있다. 첫째, 이 분야가 빠르게 변하고 있다. 둘째, 아직 아무도 정답을 찾지 못했다.

프레임워크가 난립하는 것은 초기 시장의 특징이다. 웹 프레임워크도 2010년대 초반에 같은 과정을 거쳤다. Backbone, Ember, Angular, React, Vue가 경쟁하며 생태계가 정리되기까지 5년 이상이 걸렸다. 에이전트 프레임워크는 2026년 현재 그 정리 과정의 한복판에 있다. 백화점 식품관에 가면 시식 코너가 줄지어 있다. 전부 맛보면 배가 부르지만 저녁은 해결되지 않는다. 프레임워크도 마찬가지다. 전부 시도하면 시간은 가지만 제품은 나오지 않는다.

각 프레임워크를 하나씩 살펴보되, 실전에서 쓸 때의 장단점에 집중하겠다.

## Claude Agent SDK: 하네스 그 자체를 공개하다

Claude Code의 에이전트 루프를 구동하던 하네스가 2025년 9월 SDK로 공개됐다. 원래 이름은 Claude Code SDK였지만, 이 하네스가 대부분의 주요 에이전트 루프를 구동하게 되면서 Claude Agent SDK로 이름이 바뀌었다. 열쇠를 만들어 쓰다가, 이 열쇠가 대부분의 자물쇠를 여는 것을 발견하고 공개한 셈이다.

설계 철학이 다른 프레임워크와 뚜렷하게 다르다. 비즈니스 로직을 최소화했다. 모델이 대부분을 결정한다. Anthropic의 Claude 모델에 최적화되어 있으며, 코딩 에이전트나 범용 에이전트를 만들 때 적합하다. 반대로, Claude 이외의 모델을 섞어 써야 하는 환경에서는 선택지가 아니다.

## LangChain과 LangGraph: 가장 큰 생태계의 빛과 그림자

LangChain은 에이전트 프레임워크의 대명사다. 초기에는 체인(chain), 즉 순차적 LLM 호출을 연결하는 구조였다. 프롬프트, LLM 호출, 파서, 출력을 일직선으로 잇는 방식이다. 이 구조가 복잡한 분기와 루프를 감당하지 못하면서 LangGraph로 진화했다. 체인에서 그래프로의 전환이다. 상태 머신(state machine) 기반으로 분기, 루프, 조건부 실행이 가능해졌다.

도로 체계에 비유하면 LangChain이 일방통행 도로라면, LangGraph는 교차로와 로터리가 있는 도시 도로망이다. 목적지까지 가는 경로가 여러 개 있고, 교통 상황에 따라 우회할 수 있다.

이 표에서 LangChain/LangGraph의 최대 강점은 모델 지원의 다양성이다. Claude, GPT, Llama를 섞어 쓰는 환경에서 선택지가 된다. 문서가 풍부하고 커뮤니티가 크다는 것도 장점이다.

그러나 비판도 있다. 추상화 레이어가 두껍다. LLM 호출 하나를 하려면 프레임워크 내부의 여러 계층을 거치는데, 에러가 발생하면 그 계층 안쪽에서 무슨 일이 벌어지는지 파악하기 어렵다. 자동차 엔진에 문제가 생겼는데, 보닛이 용접되어 있어 열 수 없는 상황과 비슷하다.

## CrewAI: 역할 기반의 직관성과 함정

CrewAI는 멀티에이전트를 역할 기반으로 구성한다. 리서처, 작가, 편집자라는 역할을 부여하고, 태스크를 순서대로 수행시킨다. *리서처가 조사하고, 작가가 쓰고, 편집자가 다듬는다*는 비유가 바로 코드가 된다.

직관성이 최대 장점이다. 프로토타입을 빠르게 만들 수 있다. CrewAI 팀은 *20억 워크플로우에서 배운 교훈*이라는 제목으로 실전 운영 경험을 공유한 바 있다. 대규모 사용 데이터에서 도출한 패턴이므로 참고할 가치가 충분하다.

그러나 앞 글에서 다뤘던 멀티에이전트의 함정이 여기에 해당한다. 역할 기반으로 나누면 에이전트들이 같은 컨텍스트를 공유해야 하는 상황이 발생하고, 그 순간 정보 손실과 충돌이 생긴다. Cognition의 비판이 정확히 이 지점을 겨냥한다. 영화 제작에 비유하면, 감독과 촬영감독과 편집자가 각자 다른 버전의 대본을 보고 일하면 결과물이 하나로 합쳐지지 않는 것과 같다.

## 나머지 선택지: AutoGen, PydanticAI, SmolAgents

Microsoft AutoGen은 에이전트들이 서로 대화하면서 문제를 해결하는 패턴이다. Microsoft 생태계, 특히 Azure OpenAI에 최적화되어 있다. 대화형 에이전트가 필요하고 Azure를 쓰는 환경이라면 고려할 만하다.

PydanticAI는 Python의 타입 시스템을 에이전트에 적용했다. 도구의 입출력이 타입으로 보장된다. 프로덕션 환경에서 예측 불가능한 출력이 시스템을 깨뜨리는 것을 방지한다. 타입 안전이 중요한 금융, 의료, 인프라 영역에서 적합하다. 건축의 설계도면에 치수가 정확히 표기되어 있어야 시공 오류가 줄어드는 것과 같은 원리다.

SmolAgents는 HuggingFace의 경량 프레임워크다. 가볍고 단순하며 오픈소스 모델과 잘 통합된다. 간단한 에이전트를 빠르게 만들 때 적합하다.

## 프레임워크 없이 만드는 것이 답일 수 있다

여기서 가장 중요한 이야기를 해야 한다. Anthropic의 Schluntz와 Zhang이 2024년에 이렇게 말했다.

> **핵심**: 프레임워크는 출발점으로 좋지만, 프로덕션에서는 기본 API로 직접 만드는 것이 더 유연하다.

프레임워크의 문제는 세 가지다. 추상화 레이어가 디버깅을 어렵게 만든다. 프레임워크의 제약에 맞춰야 해서 원하는 기능을 구현하지 못하는 경우가 발생한다. 빠르게 변하는 분야라 버전이 올라갈 때마다 코드가 깨진다. 건물의 내장재가 규격화되면 시공은 빠르지만, 규격에 없는 구조를 만들려면 오히려 더 많은 공사가 필요한 것과 같다.

실전에서 많이 보이는 패턴이 있다. 1단계에서 LangChain이나 CrewAI로 프로토타입을 만들고, 2단계에서 프로덕션으로 갈 때 Anthropic API나 OpenAI API로 재작성한다. 프레임워크는 학습 도구로 쓰고, 프로덕션은 직접 구현하는 것이다.

이 패턴이 확산되면서 2026년의 트렌드가 형성됐다.

왜 이런 패턴이 생기는지를 좀 더 짚어보자. 프레임워크의 가치는 *반복되는 패턴을 추상화*하는 데 있다. 웹 프레임워크가 HTTP 요청 파싱, 라우팅, 세션 관리를 추상화하듯이, 에이전트 프레임워크는 LLM 호출, 도구 실행, 루프 관리를 추상화한다. 문제는 에이전트 분야가 아직 *어떤 패턴이 반복되는지* 합의에 이르지 못했다는 점이다. 추상화할 대상이 매달 바뀌면 추상화 자체가 부채가 된다. 건물의 기초가 계속 바뀌는데 내장재를 규격화해봐야 소용이 없는 셈이다.

Anthropic이 *프레임워크보다 API*를 권하는 배경에도 이 논리가 있다. API는 가장 얇은 추상화다. LLM에 메시지를 보내고 응답을 받는 것. 이 최소 단위 위에 필요한 만큼만 직접 쌓아 올리면 프레임워크의 제약에서 자유롭다.

## 2026년, 프레임워크 시장의 방향

네 가지 흐름이 동시에 진행되고 있다.

첫째, 프레임워크 통합이다. LangChain이 LangGraph로 진화하듯, 분산된 프레임워크들이 수렴하고 있다. 둘째, MCP(Model Context Protocol) 통합이다. 모든 주요 프레임워크가 MCP를 지원하게 됐다. 도구 연결의 표준이 정해지면서 프레임워크 간 차이가 줄어들고 있다. 셋째, Skills 통합이다. Anthropic 생태계에서 Skills가 프레임워크의 역할을 일부 대체하기 시작했다. 넷째, 직접 구현 트렌드다. *프레임워크 없이 API로 만든다*가 프로덕션의 기본이 되어가고 있다.

이 네 흐름을 합치면 하나의 방향이 보인다. 프레임워크는 학습과 프로토타입의 도구로 남고, 프로덕션은 표준 프로토콜(MCP) 위에서 직접 구현하는 구조로 수렴하고 있다. 프레임워크를 고르는 것보다 MCP와 Skills를 이해하는 것이 더 긴 투자 수명을 가진다는 뜻이다.

다음 글에서는 에이전트가 텍스트를 넘어 화면을 직접 조작하는 기술을 다룬다. CLI 도구의 한계를 넘어서는 이야기다.
