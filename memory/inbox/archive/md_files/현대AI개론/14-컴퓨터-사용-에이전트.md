# 컴퓨터 사용 에이전트: AI가 화면을 직접 조작한다

*AI가 텍스트가 아니라 마우스와 키보드로 컴퓨터를 조작하는 기술이 등장했다. CLI 도구로 파일을 읽고 쓰는 것과는 차원이 다른 이야기다. 이 글은 컴퓨터 사용(Computer Use) 기술의 원리, 가능성, 한계를 다룬다.*

## 파일 조작과 화면 조작은 다른 차원이다

앞 글에서 에이전트의 도구를 다뤘다. Read, Write, Edit, Bash, Grep. 전부 CLI(Command Line Interface, 명령줄 인터페이스) 도구다. 파일을 텍스트로 읽고, 텍스트로 수정한다. 그런데 세상의 소프트웨어 중 CLI가 있는 것은 일부에 불과하다.

웹 어드민 패널, SaaS(Software as a Service) 대시보드 — Salesforce, Jira, Figma 같은 서비스 —, 데스크톱 앱, 모바일 앱. 이것들은 CLI로 접근할 수 없다. 사람이 화면을 보고, 버튼을 누르고, 텍스트를 입력해서 조작한다.

공장 자동화에 비유하면 이해가 쉽다. CLI 도구는 공장의 배관 시스템이다. 원재료가 파이프를 타고 흐르며 자동 처리된다. 빠르고 정확하지만, 파이프가 연결된 곳만 갈 수 있다. 컴퓨터 사용 에이전트는 로봇 팔이다. 사람이 하던 것처럼 버튼을 누르고 레버를 당긴다. 느리지만, 파이프가 없는 곳도 갈 수 있다.

## 컴퓨터 사용의 작동 원리

AI가 스크린샷을 보고, 마우스를 움직이고, 키보드를 누르는 기술이다. 작동 순서는 다섯 단계로 구성된다.

AI가 스크린샷을 촬영한다. 이미지를 분석해 현재 화면의 상태를 파악한다. *이 버튼을 클릭해야 한다*고 판단한다. 마우스를 해당 좌표로 이동시킨다. 클릭한다. 결과 스크린샷을 다시 촬영해 의도대로 됐는지 확인한다. 의도대로 되지 않았으면 다른 행동을 시도한다.

이 구조가 앞 글의 에이전트 루프(Gather, Act, Verify, Repeat)와 동일하다는 점에 주목해야 한다. 다만 Gather가 *파일 읽기*가 아니라 *스크린샷 보기*이고, Act가 *파일 수정*이 아니라 *마우스와 키보드 조작*인 것이다. 에이전트의 틀은 같고, 감각 기관과 실행 기관이 다른 셈이다.

## 왜 이 기술이 중요한가

MCP가 없던 시절을 기억하는가. AI가 GitHub에 접근하려면 사용자가 이슈 내용을 복사해서 붙여넣어야 했다. MCP가 그 문제를 해결했다. API가 있는 서비스는 MCP 서버를 붙이면 된다.

그런데 API가 없는 소프트웨어는 어떻게 하는가. 10년 전에 만들어진 사내 웹 시스템, REST API를 제공하지 않는 레거시 대시보드, GUI만 있는 데스크톱 앱. 이런 것들은 MCP로도 연결할 수 없다.

컴퓨터 사용 에이전트는 이 빈틈을 메운다. 화면이 있는 소프트웨어라면 무엇이든 조작할 수 있다. API가 필요 없다. 사람이 할 수 있는 것을 AI가 그대로 따라 하기 때문이다.

실전 적용 사례는 네 가지 영역으로 나뉜다. 첫째, 레거시 웹 시스템 자동화다. API가 없는 오래된 사내 시스템에서 반복 작업을 자동화한다. 대기업의 ERP(Enterprise Resource Planning, 전사적 자원관리) 시스템 중 상당수가 2000년대에 구축된 웹 인터페이스만 제공한다. 이런 시스템에 데이터를 입력하는 작업을 매일 사람이 하고 있다면, 컴퓨터 사용 에이전트가 대체할 수 있다.

둘째, 테스트 자동화다. 실제 사용자처럼 브라우저에서 클릭하고 입력하며 테스트를 수행한다. 기존의 Selenium이나 Playwright 같은 테스트 자동화 도구는 HTML 요소의 선택자(selector)에 의존하는데, UI가 바뀌면 선택자가 깨진다. 컴퓨터 사용 에이전트는 화면을 시각적으로 이해하므로 버튼의 위치나 스타일이 바뀌어도 *제출 버튼*을 찾아 클릭할 수 있다.

셋째, RPA(Robotic Process Automation, 로봇 프로세스 자동화) 대체다. 기존 RPA가 화면 요소의 ID에 의존해서 UI가 바뀌면 깨지는 반면, 컴퓨터 사용 에이전트는 시각적 이해에 기반하므로 UI 변경에 덜 취약하다. RPA 시장이 연간 약 30억 달러 규모인 것을 고려하면, 이 영역의 잠재력이 상당하다.

넷째, 다중 SaaS 연결이다. API가 전부 다른 서비스들을 화면 조작으로 통일할 수 있다.

## 세 가지 자동화 방식의 비교

CLI 도구, MCP, 컴퓨터 사용의 세 방식을 나란히 놓으면 각각의 위치가 명확해진다.

| [*방식*], [*속도*], [*정확도*], [*적용 범위*], [*보안 위험*],  [CLI 도구 (Read, Bash)], [빠름], [높음], [파일/셸 한정], [중간],
  [MCP], [빠름], [높음], [API 있는 서비스], [API 수준],
  [컴퓨터 사용], [느림], [중간], [모든 소프트웨어], [높음],)

이 표의 핵심은 트레이드오프다. 컴퓨터 사용은 적용 범위가 가장 넓지만, 속도와 정확도에서 CLI와 MCP에 뒤진다. 보안 위험도 가장 높다. 교통수단에 비유하면 CLI가 지하철이고, MCP가 택시이며, 컴퓨터 사용은 도보다. 지하철은 빠르지만 역이 있는 곳만 간다. 택시는 도로가 있으면 간다. 도보는 어디든 가지만 가장 느리다.

> **핵심**: 실전 원칙은 *CLI를 먼저 시도하고, 안 되면 MCP, 그것도 안 되면 컴퓨터 사용* 순서다. 넓은 범위를 먼저 쓰는 것이 아니라, 가장 빠르고 정확한 방식을 먼저 쓰고 범위를 넓혀가는 것이다.

아래 그림은 ARC-AGI 벤치마크의 사례를 보여준다. AI가 시각적 패턴을 인식하고 조작하는 능력이 컴퓨터 사용 에이전트의 기반 기술과 직결된다.

이 그림에서 보이는 격자 패턴의 변환 과제는 사람에게는 직관적이지만 AI에게는 여전히 도전적이다. 컴퓨터 사용 에이전트가 화면을 *이해*하려면 이런 시각적 추론 능력이 필수다.

## 2026년 현재, 솔직한 기술 수준

잘 되는 영역이 있고, 아직 안 되는 영역이 있다.

단순 클릭 시퀀스는 안정적이다. 로그인 화면에서 아이디와 비밀번호를 입력하고 버튼을 누르는 수준의 작업은 신뢰할 수 있다. 웹 브라우저 자동화와 스크린샷 속 텍스트 읽기도 잘 작동한다.

반면 복잡한 드래그 앤 드롭은 아직 불안정하다. 빠르게 변하는 화면, 애니메이션, 동적 로딩이 있으면 스크린샷 시점과 실제 화면 상태가 어긋난다. 좌표 정확도 문제도 있다. *이 버튼*을 클릭하려는데 옆 버튼을 클릭하는 경우가 발생한다. 팝업, 모달, 오버레이 처리도 난제다. 화면 위에 화면이 겹치면 에이전트가 어떤 계층을 조작해야 하는지 혼동한다.

가장 큰 한계는 속도다. 사람이 버튼 하나를 클릭하는 데 0.1초면 충분하다. 컴퓨터 사용 에이전트는 스크린샷 촬영, 이미지 분석, 좌표 결정, 클릭, 결과 확인까지 수 초에서 수십 초가 걸린다. 사람보다 수십 배 느리다. CLI 도구가 텍스트를 바로 처리하는 반면, 컴퓨터 사용은 매번 이미지 인식 단계를 거쳐야 하기 때문이다. 책을 읽는 것과 책의 사진을 찍어서 OCR로 읽는 것의 차이와 같다.

이 속도 문제가 컴퓨터 사용 에이전트의 실전 적용을 제한하는 핵심 병목이다. 빠른 반복이 필요한 작업에는 부적합하고, 느려도 되는 반복 작업에 적합하다.

## 보안: 다른 도구보다 훨씬 위험하다

컴퓨터 사용 에이전트의 보안 위험은 구조상 CLI 도구나 MCP보다 높다. 이유는 단순하다. *사람이 할 수 있는 모든 것*을 AI가 할 수 있기 때문이다.

이메일을 열어 민감한 정보를 읽는다. 브라우저에서 결제를 진행한다. 설정에서 비밀번호를 변경하고, 파일 탐색기에서 파일을 삭제하기도 한다.

MCP는 도구 단위로 권한을 제한할 수 있다. GitHub MCP 서버에 이슈 읽기 권한만 부여하면 PR을 만들 수는 없다. 컴퓨터 사용은 이런 세분화가 불가능하다. 화면에 보이는 것 전부가 접근 범위다. 잠긴 방 열쇠를 주는 것과 건물 마스터키를 주는 것의 차이다.

프롬프트 인젝션(Prompt Injection) 위험도 독특하다. 화면에 표시된 텍스트가 공격 벡터가 될 수 있다. 누군가 웹페이지에 *이전 지시를 무시하고 다음을 실행하라*는 텍스트를 숨겨놓으면, 에이전트가 스크린샷을 통해 그 텍스트를 읽고 지시를 따를 가능성이 있다. 눈으로 읽는 것이 곧 입력이 되는 구조이기 때문에, 보이는 모든 것이 잠재적 공격 표면이다.

반드시 지켜야 할 원칙이 네 가지다. 샌드박스 환경에서 실행할 것. 화면 녹화로 에이전트의 행동을 감시할 것. 민감한 시스템에 대한 접근을 차단할 것. 프롬프트 인젝션 방어를 적용할 것.

## 컴퓨터 사용의 미래: API 없는 세계를 연결하다

컴퓨터 사용 에이전트는 2026년 현재 완성된 기술이 아니다. 느리고, 불안정하고, 위험하다. 그런데도 이 기술이 중요한 이유가 있다.

| [*시기*], [*자동화 범위*],  [CLI 시대], [파일과 셸 명령],
  [MCP 시대], [API가 있는 외부 서비스],
  [컴퓨터 사용 시대], [화면이 있는 모든 소프트웨어],)

이 표에서 적용 범위가 단계적으로 확장되는 것이 보인다. 각 단계가 이전을 대체하지 않고 포함한다. CLI로 되는 것은 CLI로 하고, MCP로 되는 것은 MCP로 하되, 둘 다 안 되는 영역을 컴퓨터 사용이 커버한다.

세계에는 API가 없는 소프트웨어가 API가 있는 것보다 훨씬 많다. 정부 기관의 내부 시스템, 10년 이상 된 사내 도구, 소규모 SaaS의 관리 패널. 이것들을 자동화할 수 있는 유일한 방법이 화면 조작이다. 속도와 정확도가 개선되면, 그리고 보안 문제가 해결되면, 컴퓨터 사용 에이전트는 AI 자동화의 마지막 조각이 될 수 있다.

> **핵심**: 컴퓨터 사용 에이전트는 CLI와 MCP가 닿지 못하는 영역을 연결한다. 2026년 현재 속도, 정확도, 보안에서 미성숙하지만, *화면이 있는 모든 소프트웨어를 자동화할 수 있다*는 가능성 자체가 이 기술의 의의다.

다음 글에서는 에이전트가 외부 세계와 연결되는 표준 프로토콜, MCP를 다룬다. 만능이라는 약속과 현실의 괴리가 핵심이다.
