# Skills: 도구가 있어도 잘 쓸 줄 모른다

*MCP가 에이전트에게 외부 세계의 도구를 연결했다. 그런데 도구가 있다고 일을 잘 하는 것은 아니다. Skills는 에이전트에게 전문 지식과 절차를 패키지로 제공한다. 이 글은 Skills의 구조와 작성법을 실전 수준으로 다루고, 왜 Simon Willison이 Skills가 MCP보다 더 큰 변화일 수 있다고 말했는지를 해부한다.*

## MCP가 채우지 못한 빈자리

앞 글에서 MCP의 세 가지 문제를 다뤘다. 토큰 과소비, 보안 취약점, 구현 복잡성. 그런데 더 근본에 해당하는 문제가 하나 있다.

MCP는 *어떤 도구에 접근할 수 있는가*를 알려준다. GitHub API를 호출한다. Slack 메시지를 읽는다. DB에 쿼리를 날릴 수 있다. 그런데 이것을 안다고 일을 잘 하는 것은 아니다.

병원에 비유하면 차이가 명확해진다. MCP는 *여기 청진기, MRI(자기공명영상), 혈액검사 장비가 있다*고 알려주는 것이다. 장비 목록이다. Skills는 *이 환자에게 이런 증상이 있으니까 먼저 혈액검사를 하고, 결과에 따라 MRI를 찍고, 이 결과가 나오면 이 전문의에게 의뢰하라*는 진료 프로토콜이다. 장비를 아는 것과 진료를 잘 하는 것은 완전히 다르다.

실전 예시로 보자. AI에게 MCP로 GitHub 도구를 줬다. 그런데 AI는 이 프로젝트에서 PR(Pull Request) 제목을 어떤 형식으로 올려야 하는지, 어떤 체크리스트를 거쳐야 하는지, 리뷰어를 누구에게 지정해야 하는지를 모른다. 도구는 있는데 사용법을 모르는 것이다. 결과물의 형식이 맞지 않고, 팀의 관례를 무시하고, 매번 사람이 수정해야 한다. 도구 접근권과 전문성의 격차가 여기서 드러난다.

## Skills의 탄생: 마크다운 한 장이 바꾼 것

2025년 10월 16일, Anthropic이 Agent Skills를 오픈 스탠다드로 공개했다. agentskills.io가 공식 사이트다.

같은 날, Simon Willison이 한마디를 남겼다. *Skills가 MCP보다 더 큰 변화일 수 있다(Skills might be a bigger deal than MCP).* MCP를 *AI의 USB-C*라고 명명한 바로 그 사람이다. 그가 MCP보다 Skills를 더 크게 평가한 이유를 이해하려면 Skills의 구조를 직접 봐야 한다.

Skills의 실체는 놀라울 정도로 단순하다. 마크다운 파일이다. 프로젝트 디렉터리 안에 `.claude/skills/pr-review/` 같은 폴더를 만들고, 그 안에 SKILL.md를 넣는다. JSON-RPC 프로토콜도 없고, 서버 구현도 없고, OAuth도 없다.

이것이 MCP와의 결정적 차이다. MCP 서버를 만들려면 JSON-RPC, 전송 계층, 인증까지 엔지니어가 필요하다. Skills를 만드는 데는 마크다운 파일을 작성할 수 있는 사람이면 충분하다. 프로그래머가 아닌 프로젝트 관리자, 디자이너, 도메인 전문가도 Skills를 만들 수 있다. 도구 생태계의 민주화인 셈이다.

Willison이 Skills를 더 크게 본 이유가 여기에 있다. MCP는 개발자만 만들 수 있다. Skills는 전문 지식을 가진 사람이면 누구나 만들 수 있다. 의사가 진료 프로토콜을 스킬로 작성하고, 변호사가 계약서 검토 절차를 스킬로 만들고, 마케터가 브랜드 가이드라인을 스킬로 패키징한다. 지식 노동의 모든 영역에서 전문성을 재사용 가능한 패키지로 만들 수 있다는 뜻이다.

## SKILL.md 해부: 프론트매터부터 본문까지

스킬의 실체를 해부하자. 하나의 SKILL.md 파일은 두 부분으로 구성된다. YAML 프론트매터와 마크다운 본문이다.

아래는 실제로 작동하는 커밋 스킬이다. 각 줄을 해설한다.

```
---
name: commit                          # 슬래시 커맨드 이름. /commit으로 호출
description: 커밋 전 테스트와 린트를    # Claude가 자동 매칭에 사용하는 설명.
  자동 실행하고 컨벤셔널 커밋 형식으로   # 이 텍스트가 작업과 얼마나 관련 있는지로
  메시지를 작성한다                     # 로드 여부가 결정된다
disable-model-invocation: true         # true면 사용자만 호출 가능.
                                       # 부작용 있는 작업에 필수
allowed-tools: Bash(npm *), Bash(git *)  # 이 스킬 활성 시 허용할 도구 목록
---

## 절차

1. `npm test` 실행
2. 실패 시 원인 분석 후 수정, 재실행
3. `npm run lint` 실행
4. 통과 시 변경 내용을 분석하여 커밋 메시지 작성
5. Conventional Commits 형식 준수: type(scope): description
6. `git commit` 실행
```

프론트매터의 각 필드가 하는 역할을 정리하면 다음과 같다.

| 필드 | 역할 | 기본값 |
| --- | --- | --- |
| `name` | 슬래시 커맨드 이름. 소문자, 숫자, 하이픈만 허용. 최대 64자 | 디렉터리명 |
| `description` | Claude가 자동 매칭에 사용하는 설명. 스킬의 존재 이유 | 본문 첫 문단 |
| `disable-model-invocation` | `true`면 사용자만 호출 가능. 배포, 전송 등 부작용 있는 작업에 필수 | `false` |
| `user-invocable` | `false`면 슬래시 메뉴에서 숨김. Claude만 자동 호출. 배경 지식용 | `true` |
| `allowed-tools` | 스킬 활성 시 승인 없이 사용 가능한 도구 목록 | 없음 |
| `context` | `fork`면 독립 서브에이전트에서 실행. 별도 컨텍스트 윈도우 | `inline` |
| `agent` | 포크 시 서브에이전트 타입. Explore, Plan, general-purpose 중 선택 | `general-purpose` |
| `model` | 스킬 실행 시 사용할 모델 지정. `inherit`면 부모 세션 모델 사용 | 세션 기본 모델 |
| `paths` | gitignore 패턴으로 조건부 활성화. 특정 경로 작업 시에만 로드 | 없음 |

이 표에서 가장 중요한 필드는 `description`이다. 나머지는 선택이지만, description은 스킬의 존재 이유다. Claude가 이 텍스트를 보고 *지금 이 스킬이 필요한가*를 판단한다. description이 모호하면 스킬은 영원히 호출되지 않는다.

본문은 자유 형식이다. 절차를 적어도 되고, 규칙을 나열해도 되고, 예시를 보여줘도 된다. 핵심은 Claude에게 *어떻게 해야 하는지*를 가르치는 것이다. 500줄을 넘기면 지원 파일로 분리하는 것을 권장한다.

## 스킬 작성 실전: 좋은 스킬 vs 나쁜 스킬

스킬을 만들 수 있다는 것과 잘 만드는 것은 다르다. 나쁜 스킬과 좋은 스킬의 차이를 구체적으로 보자.

나쁜 스킬의 첫 번째 안티패턴은 *과적(overloading)*이다. 하나의 스킬에 코드 리뷰, 테스트 작성, 문서 생성, 배포까지 넣는 경우다. 800줄짜리 스킬은 Claude의 컨텍스트를 과도하게 점유하고, 무엇이 핵심인지 흐려진다. 한 스킬 = 한 워크플로가 원칙이다.

두 번째 안티패턴은 *MCP 미러링*이다. MCP 서버의 전체 API를 스킬로 복제하는 것이다. *GitHub API의 모든 엔드포인트 사용법*을 스킬에 적는 것은 API 문서를 컨텍스트에 덤프하는 것과 같다. MCP는 게이트웨이, 스킬은 프로토콜이다. 스킬에는 *이 프로젝트에서 GitHub를 어떻게 쓰는가*를 적어야 한다.

세 번째 안티패턴은 *자동 호출 남용*이다. 부작용이 있는 스킬을 자동 호출 가능하게 두는 것이다. 배포 스킬, 메시지 전송 스킬이 자동으로 트리거되면 의도하지 않은 배포가 일어날 수 있다. 부작용이 있으면 `disable-model-invocation: true`는 선택이 아니라 필수다.

좋은 스킬의 특징은 명확하다. 설명이 구체적이고, 절차가 단계별로 나뉘어 있고, 허용 도구가 최소한으로 제한되어 있고, 검증 단계가 포함되어 있다. 아래는 좋은 스킬의 두 번째 예시다.

```
---
name: deep-research
description: 코드베이스를 전수 탐색하여 특정 주제에 대한 심층 분석 보고서를 작성한다
context: fork
agent: Explore
allowed-tools: Read, Grep, Glob
---

$ARGUMENTS에 대해 다음 절차를 수행하라.

1. Glob으로 관련 파일 패턴 탐색
2. Grep으로 핵심 키워드 검색
3. 발견된 파일을 Read로 분석
4. 파일 경로와 줄 번호를 포함한 보고서 작성
5. 발견 사항을 중요도 순으로 정렬
```

이 스킬이 좋은 이유는 세 가지다. `context: fork`로 메인 세션을 오염시키지 않는다. `agent: Explore`로 읽기 전용 도구만 사용한다. 절차가 구체적이어서 Claude가 무엇을 해야 하는지 모호하지 않다.

## 자동 매칭: 부르지 않아도 알아서 온다

Skills의 핵심 혁신 중 하나는 자동 매칭이다. 사용자가 스킬 이름을 알 필요가 없다.

작동 방식은 이렇다. 사용자가 *PR 만들어줘*라고 말한다. Claude가 설치된 스킬들의 description을 스캔한다. commit-skill은 *커밋 메시지 작성*이라 관련도가 낮다. pr-review는 *PR 리뷰 절차와 기준*이라 관련도가 높다. deploy-skill은 *배포 프로세스*라 관련도가 낮다. Claude가 자동으로 pr-review 스킬을 선택하고 본문을 로드한다.

이 과정에서 description만 스캔한다는 것이 핵심이다. 20개 스킬이 설치되어 있어도 평소 비용은 수백 토큰에 불과하다. 관련 스킬이 선택된 후에야 본문이 로드되고, 본문에서 참조하는 지원 파일은 필요할 때만 추가 로드된다. 3단계 점진적 공개다.

> **핵심**: *자연스럽게 말하면 자동으로 작동한다*가 Skills의 설계 원칙이다. MCP는 백화점 카탈로그를 건네주고 *골라*라고 하는 것이고, Skills는 비서가 *지금 이게 필요하실 것 같습니다*라고 꺼내오는 것이다.

자동 매칭이 실패하는 경우도 있다. description에 사용자가 실제로 쓰는 단어가 없으면 매칭이 안 된다. *PR 리뷰 절차*라고 적었는데 사용자가 *코드 검토해줘*라고 말하면 매칭 확률이 떨어진다. 해결책은 description에 사용자가 자연스럽게 사용할 키워드를 포함시키는 것이다. *코드 리뷰, PR 검토, 풀리퀘스트 리뷰 절차와 기준*처럼 동의어를 나열하면 매칭률이 올라간다.

스킬 description의 총합이 컨텍스트 윈도우의 2%를 초과하면 일부 스킬이 자동으로 제외된다. 기본 한도는 16,000자다. 이것은 앞 글에서 다뤘던 컨텍스트 엔지니어링의 Select 축이 프로토콜 수준에서 구현된 것이다.

## 토큰 경제학: Skills가 MCP를 이기는 지점

앞 글에서 MCP의 토큰 과소비 문제를 다뤘다. MCP 서버를 연결하면 전체 도구 스키마가 컨텍스트에 상주한다. 쓰든 안 쓰든 수천 토큰이 고정 점유된다.

Skills는 이 문제를 점진적 공개(Progressive Disclosure)로 해결한다. 작동 방식이 세 단계로 나뉜다.

| 단계 | 내용 | 토큰 비용 |
| --- | --- | --- |
| 1단계: 메타 스캔 | YAML 프론트매터의 이름과 설명만 읽음 | 스킬당 수십 토큰 |
| 2단계: 본문 로드 | 이 스킬이 지금 필요하다고 판단되면 본문 로드 | 수백~수천 토큰 |
| 3단계: 참조 파일 로드 | 세부 내용이 필요하면 체크리스트, 예시, 스크립트 등 추가 로드 | 필요한 만큼 |

도서관에 비유하면 이해가 쉽다. MCP는 도서관의 모든 책을 책상 위에 올려놓는 것이다. 37권이 책상을 덮고 있으면 필요한 책을 찾기도, 펼쳐 읽기도 어렵다. Skills는 서가 목록(1단계)만 훑어보고, 필요한 책(2단계)을 꺼내 오고, 특정 챕터(3단계)만 펼치는 것이다. 책상 위에는 항상 지금 읽는 책만 있다.

## 스킬 vs 프롬프트 vs MCP: 세 가지 확장 방식의 비교

에이전트의 능력을 확장하는 방식은 크게 세 가지다. 프롬프트(CLAUDE.md 등 메모리 파일), MCP, Skills. 이 셋을 혼동하는 경우가 많다. 핵심 차이를 표로 정리한다.

| 항목 | 프롬프트 (CLAUDE.md) | MCP | Skills |
| --- | --- | --- | --- |
| 본질 | 매 세션 주입되는 지시문 | 외부 도구 연결 프로토콜 | 전문 지식과 절차의 패키지 |
| 부여하는 것 | 배경 맥락 | 행동 범위 | 행동 품질 |
| 토큰 비용 | 매번 전체 로드 | 연결 시 전체 스키마 상주 | JIT 3단 로드 |
| 로드 시점 | 세션 시작 시 항상 | 서버 연결 시 항상 | 관련 작업 감지 시에만 |
| 제작 난이도 | 마크다운 작성 | JSON-RPC 서버 구현 | 마크다운 작성 |
| 재사용 단위 | 프로젝트 단위 | 서버 단위 | 워크플로 단위 |
| 자동 매칭 | 없음 (항상 로드) | 없음 (항상 노출) | 있음 (description 기반) |
| 실행 코드 포함 | 불가 | 서버 측 구현 | 스크립트 번들링 가능 |

이 표에서 가장 중요한 차이는 *로드 시점*이다. CLAUDE.md는 매 세션마다 전부 로드된다. 100줄이든 500줄이든 매번 컨텍스트를 점유한다. MCP는 연결된 서버의 전체 도구 스키마가 항상 상주한다. Skills만 *관련 있을 때만* 로드된다.

그래서 세 가지의 용도가 다르다. 프로젝트 전반에 항상 적용되는 규칙(코딩 컨벤션, 응답 언어, 금지 사항)은 CLAUDE.md에 적는다. 외부 시스템과의 연결(GitHub, Slack, DB)은 MCP로 처리한다. 특정 작업의 전문 절차(PR 리뷰, 배포, 커밋, 코드 생성)는 Skills로 패키징한다.

## MCP와 Skills의 분업: 행동 범위 대 행동 품질

MCP가 죽었다는 이야기가 아니다. 역할이 다르다.

*GitHub에서 이슈 가져와서 PR 만들어줘.* 이 요청을 처리하려면 둘 다 필요하다. MCP가 하는 일은 GitHub API에 접속하고, 이슈 내용을 가져오고, PR 생성 API를 호출하는 것이다. Skills가 하는 일은 이 프로젝트에서 PR 제목 형식이 무엇인지, 리뷰어를 몇 명 지정해야 하는지, 테스트 커버리지 기준이 무엇인지를 알려주는 것이다.

| 항목 | MCP | Skills |
| --- | --- | --- |
| 본질 | 외부 도구 연결의 표준 프로토콜 | 전문 지식과 절차의 패키지 |
| 부여하는 것 | 행동 범위 (무엇에 접근 가능한가) | 행동 품질 (어떻게 잘 수행하는가) |
| 기술 구조 | JSON-RPC 2.0, 3자 구조 | 마크다운 + YAML + 선택적 스크립트 |
| 토큰 비용 | 연결 시 전체 도구 스키마 상주 | JIT 로드, 평소 수백 토큰 |
| 제작 난이도 | 엔지니어 필요 | 누구나 가능 |
| 2026년 역할 | 외부 시스템의 보안 게이트웨이 | 에이전트 내부의 절차적 전문지식 |

이 표에서 가장 중요한 행은 *부여하는 것*이다. MCP는 *무엇에 접근할 수 있는가*를 제공한다. Skills는 *어떻게 잘 수행하는가*를 제공한다. 둘은 대체가 아니라 보완 관계다. MCP로 행동 범위를 확장하고, Skills로 행동 품질을 높인다.

2026년의 추세는 분업이 명확해지는 방향이다. 초기에 MCP가 *만능 API 미러*로 남용되던 패턴에서 벗어나, MCP는 보안 게이트웨이 역할에 집중하고 Skills가 실질적 전문화를 담당하는 구조로 수렴하고 있다. Shrivu Shankar는 이 방향을 이렇게 요약했다. *MCP는 보안 게이트웨이여야 하지, 포괄적 API 미러가 아니다.*

## Before/After: 스킬이 만드는 품질 차이

추상적인 이야기를 그만두고 구체적 차이를 보자. 같은 요청을 스킬 없이 처리했을 때와 스킬이 있을 때 결과가 어떻게 달라지는가.

요청: *이 변경사항으로 PR 만들어줘.*

스킬 없이 처리한 경우, Claude는 범용 지식으로 PR을 만든다. 제목이 *Update files*처럼 모호하다. 설명이 변경 파일 목록을 나열하는 수준이다. 리뷰어 지정이 없다. 라벨이 없다. 테스트 통과 여부 확인이 없다. 팀의 관례와 동떨어진 결과물이 나오고, 사람이 제목을 고치고, 설명을 다시 쓰고, 리뷰어를 지정하고, 라벨을 추가해야 한다.

스킬이 있는 경우, PR 스킬에 이 프로젝트의 규칙이 적혀 있다. 제목은 *Conventional Commits 형식(feat/fix/docs 등)*으로 작성된다. 설명에 변경 이유, 테스트 계획, 체크리스트가 포함된다. 리뷰어가 변경된 코드의 소유자 기준으로 자동 지정된다. 관련 라벨이 붙는다. 커밋 전에 테스트와 린트가 자동 실행된다. 사람이 할 일은 최종 확인뿐이다.

두 번째 예시도 보자. 요청: *이 코드베이스 구조를 분석해줘.*

스킬 없이 처리하면 Claude가 현재 디렉터리의 파일 몇 개를 읽고 대략적인 설명을 한다. 깊이가 얕고 놓치는 파일이 많다. 스킬이 있으면 `context: fork`로 독립 에이전트가 실행되어 Glob으로 전체 파일 패턴을 탐색하고, Grep으로 핵심 키워드를 검색하고, 발견된 파일을 체계적으로 분석한다. 메인 세션의 컨텍스트를 소비하지 않으면서 훨씬 깊은 분석이 나온다.

> **핵심**: 스킬의 가치는 *한 번의 결과물*이 아니라 *매번의 일관성*에 있다. 사람이 매번 똑같은 지시를 반복하지 않아도 프로젝트의 품질 기준이 자동으로 적용된다.

## 2026년 최신 기능: context fork와 agent 프론트매터

Skills는 빠르게 진화하고 있다. 2026년에 추가된 핵심 기능 두 가지를 짚는다.

`context: fork`(2026년 3월)는 Skills 생태계의 게임 체인저다. 스킬을 독립 서브에이전트로 실행한다. 별도 토큰 예산과 별도 컨텍스트 윈도우를 갖는다. 이전에는 무거운 스킬이 부모 컨텍스트를 잡아먹는 문제가 있었다. 코드베이스 분석 스킬이 수십 개 파일을 읽으면 메인 대화의 컨텍스트가 소진됐다. context fork 이후에는 스킬이 격리된 환경에서 실행되므로 부모 컨텍스트 오염이 0%다.

칸막이 사무실에 비유하면, 이전에는 모든 직원이 같은 책상을 공유하고 있었다. 한 직원이 자료를 잔뜩 펼쳐놓으면 다른 직원의 작업 공간이 줄어들었다. context fork 이후에는 각자 칸막이 안에서 독립적으로 일한다. 결과만 정리해서 공유 공간에 돌려보낸다.

주의할 점이 있다. `context: fork`는 명시적 작업 지시가 있는 스킬에서만 의미가 있다. *API 컨벤션을 따르라*처럼 배경 지식만 담긴 스킬을 fork로 실행하면, 서브에이전트가 지시를 받지만 수행할 작업이 없어서 빈 결과를 돌려보낸다. fork는 *작업*을 격리하는 것이지, *지식*을 격리하는 것이 아니다.

`agent` 프론트매터(2026년 3월)는 스킬이 실행될 때 어떤 종류의 서브에이전트로 동작할지 지정한다. 세 가지 선택지가 있다.

| 타입 | 용도 | 허용 도구 |
| --- | --- | --- |
| Explore | 코드베이스 탐색, 조사 | 읽기 전용 (Read, Grep, Glob) |
| Plan | 계획 수립, 분석 | 읽기 + 제한적 실행 |
| general-purpose | 범용 작업 수행 | 전체 도구 |

Explore 타입이 가장 안전하다. 파일을 읽기만 하고 수정하지 않으므로 부작용이 없다. 코드 분석, 리서치, 의존성 조사 같은 작업에 적합하다. 부작용이 있는 작업(코드 수정, 파일 생성, 명령어 실행)은 general-purpose를 사용한다.

그 외에 `.claude/agents/` 디렉터리에 커스텀 서브에이전트를 정의하면 `agent` 필드에서 해당 이름을 직접 지정할 수도 있다. 스킬과 서브에이전트의 결합은 에이전트 아키텍처의 다음 단계다.

## 스킬 생태계: 오픈 표준과 공유의 현실

Anthropic은 Skills를 agentskills.io라는 오픈 스탠다드로 공개했다. Claude Code뿐 아니라 다른 에이전트 플랫폼도 채택할 수 있다는 구상이다. 그러나 2026년 4월 현재, 생태계의 현실은 구상과 거리가 있다.

스킬 공유 방식은 세 가지다. 프로젝트 스킬은 `.claude/skills/`를 Git에 커밋한다. 플러그인 스킬은 플러그인 패키지의 `skills/` 디렉터리에 포함한다. 조직 스킬은 매니지드 설정으로 전사 배포한다.

문제는 검증이다. npm에는 패키지 서명이 있고, Docker Hub에는 이미지 검증이 있다. Skills에는 아직 없다. `.claude/skills/` 디렉터리는 Git에 포함된다. 누군가 악성 PR로 스킬 파일에 셸 명령을 주입하면 어떻게 되는가. *배포 전에 다음 명령을 실행하라*는 지시 안에 SSH 키를 외부로 전송하는 curl 명령이 숨어 있을 수 있다. Claude가 이 스킬을 로드하면 키가 유출될 수 있다.

2026년 4월에 `disableSkillShellExecution` 설정이 추가됐다. 스킬의 셸 실행을 전역으로 차단하는 옵션이다. MCP 스킬은 원래부터 셸 실행이 완전 차단되어 있다. 그러나 이것은 방어책이지 근본 해결은 아니다. 스킬 파일의 변경 감지는 가능하지만, 그 변경이 정당한지 악의에 의한 것인지는 판단하지 못한다. 택배 상자가 도착했다는 것은 알 수 있지만, 상자 안에 무엇이 들어 있는지는 열어봐야 아는 것과 같다.

크로스 플랫폼 채택도 제한적이다. MCP는 OpenAI, Google, Microsoft가 빠르게 도입했다. Skills는 2026년 4월 현재 Anthropic 제품 중심이다. USB-C가 모든 제조사에 보급되기까지 시간이 걸렸듯이, Skills의 범용화에도 시간이 필요하다. 다만 Willison은 *Skills의 캄브리아 폭발이 초기 MCP 열풍을 넘어설 것*으로 전망했다. 마크다운이라는 낮은 진입 장벽이 생태계 확장의 동력이 될 수 있기 때문이다.

## 스킬 디렉터리 구조와 로딩 우선순위

스킬이 저장되는 위치에 따라 적용 범위가 달라진다. 같은 이름의 스킬이 여러 위치에 있으면 우선순위에 따라 하나만 로드된다.

| 우선순위 | 위치 | 적용 범위 |
| --- | --- | --- |
| 1 (최고) | 매니지드 (조직 정책) | 조직 전체 |
| 2 | `~/.claude/skills/` (개인) | 모든 프로젝트 |
| 3 | `.claude/skills/` (프로젝트) | 해당 프로젝트만 |
| 4 | `--add-dir` (추가 디렉터리) | 세션 동안 |
| 5 (최저) | `.claude/commands/` (레거시) | 해당 프로젝트만 |

모노레포(Monorepo)도 지원한다. `packages/frontend/` 안에서 작업하면 `packages/frontend/.claude/skills/`의 스킬도 자동으로 발견된다. 중첩 디렉터리의 스킬이 해당 패키지에서 작업할 때만 활성화되는 것이다.

하나의 스킬 디렉터리는 다음과 같은 구조를 갖는다.

```
.claude/skills/my-skill/
├── SKILL.md           # 메타 + 지시문 (필수, 500줄 이하 권장)
├── reference.md       # 상세 API 문서 (필요 시 로드)
├── examples/
│   └── sample.md      # 기대 출력 형식
└── scripts/
    └── validate.sh    # Claude가 실행하는 스크립트
```

SKILL.md가 유일한 필수 파일이다. 나머지는 전부 선택이다. 지원 파일은 SKILL.md 본문에서 참조하면 Claude가 필요할 때 로드한다. 스크립트 번들링이 가능하다는 것은 결정론적 연산(테스트 실행, 린트 검사, 파일 변환)과 LLM 추론을 하나의 패키지로 묶을 수 있다는 뜻이다.

동적 컨텍스트 주입도 가능하다. SKILL.md 본문에 `!`셸 명령`` 문법을 쓰면 스킬 로드 시 해당 명령이 먼저 실행되고, 결과가 프롬프트에 삽입된다. Claude는 명령 자체가 아니라 실행 결과만 본다. 전처리 단계인 셈이다.

## 전체 아키텍처: 5개 계층의 관계

이 시리즈의 척추 서사를 한눈에 정리하자.

| 계층 | 역할 | 한계 → 다음 계층으로 |
| --- | --- | --- |
| 프롬프트 | 한 턴의 지시 | 한 턴만으로는 복잡한 작업 불가 |
| 컨텍스트 엔지니어링 | 상황 정보의 총체 | 수동 관리에 한계 |
| 하네스 | 컨텍스트를 자동 관리하는 소프트웨어 | 외부 세계와 연결 필요 |
| MCP | 외부 시스템 게이트웨이 (행동 범위) | 도구를 잘 쓸 줄 모름 |
| Skills | 전문성 패키지 (행동 품질) | 검증 부재, 생태계 한계 |

이 표에서 각 계층은 아래를 대체하지 않고 포함한다. 프롬프트가 쓸모없어진 것이 아니다. 프롬프트가 더 큰 그림의 한 조각으로 자리잡은 것이다.

아래 그림은 AI 활용 기술의 5개 계층을 보여준다. 프롬프트 엔지니어링에서 시작해 컨텍스트 엔지니어링, 하네스, MCP, Skills 순서로 쌓인다. 각 계층은 이전을 대체하지 않고 포함한다.

![AI 활용 기술의 5개 계층. 프롬프트 → 컨텍스트 엔지니어링 → 하네스 → MCP → Skills 순서로 각 계층이 쌓인다.](../../../03-spine.png)

이 그림을 이해하면 2026년에 AI를 쓰는 올바른 투자 방향이 보인다. 프롬프트를 다듬는 데 시간을 쓸 것이 아니라, CLAUDE.md를 잘 설계하고(컨텍스트 엔지니어링), 좋은 하네스를 선택하고(Claude Code, Cursor 등), 필요한 MCP만 최소한으로 연결하고, 프로젝트에 맞는 Skills를 만드는 것이다.

> **핵심**: Skills는 에이전트에게 *전문성*을 부여하는 재사용 가능한 지식 패키지다. MCP가 행동 범위를 확장한다면 Skills는 행동 품질을 높인다. 마크다운 파일이라는 단순한 형태가 역설적으로 강점이다. 누구나 만들 수 있고, 토큰을 아끼며, 필요할 때만 자동으로 로드된다.
