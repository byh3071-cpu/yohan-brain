# 모델을 만드는 것보다 돌리는 것이 더 비싸다

*AI 비용의 대부분은 학습이 아니라 서빙에서 발생한다. 학습은 한 번이지만 서빙은 매 요청마다 반복된다. vLLM과 PagedAttention이 이 비용 구조를 근본적으로 바꿨다.*

## 학습 비용은 한 번이고, 서빙 비용은 영원하다

많은 사람이 AI 비용을 학습 비용과 동일시한다. GPT-4의 학습에 1억 달러 이상이 들었다는 보도가 이 인식을 강화했다. 그러나 이 인식은 절반만 맞다.

학습은 한 번이다. 모델을 만들 때 한 번 돈을 쓴다. 서빙, 즉 추론(inference)은 매 요청마다 돈이 든다. 사용자가 100만 명이고 하루 평균 10번씩 쓰면 하루 1,000만 건의 추론이 발생한다. 1년이면 36억 건이다.

식당을 비유로 쓰자. 레시피를 개발하는 비용은 한 번이다. 그러나 매일 수천 명의 손님에게 요리를 내놓는 비용은 매일 반복된다. 장기적으로 보면 레시피 개발비보다 매일의 운영비가 압도적으로 크다. AI 서빙도 같은 구조다. 모델이 대중화될수록 학습 비용 대비 서빙 비용의 비율이 기하급수적으로 커진다.

이 때문에 추론 인프라(inference infrastructure)가 AI 산업의 핵심 경쟁 영역이 됐다. 같은 모델이라도 어떤 인프라 위에서 돌리느냐에 따라 비용이 2~4배 달라진다.

## KV-Cache: 컨텍스트가 길면 비싼 이유

트랜스포머 모델이 토큰을 생성하는 과정을 이해해야 비용 구조가 보인다. 트랜스포머는 새 토큰을 만들 때마다 이전에 생성된 모든 토큰을 참조한다. 구체적으로는 각 토큰의 Key 벡터와 Value 벡터를 참조한다. Key는 *내가 어떤 정보를 갖고 있는가*를 나타내는 벡터이고, Value는 *내가 제공할 수 있는 구체적 정보*를 담은 벡터다. 이 두 벡터를 합쳐서 KV(Key-Value)라고 부른다.

KV-Cache(KV 캐시)는 이 KV 벡터들을 메모리에 저장해두는 것이다. 새 토큰을 만들 때마다 이전 토큰의 KV를 처음부터 다시 계산하면 극도로 비효율적이다. 한 번 계산한 KV를 캐시에 저장해두면 재계산 없이 바로 참조할 수 있다. 수학 시험에서 중간 계산 결과를 여백에 적어두는 것과 같다. 매번 처음부터 다시 푸는 대신 적어둔 결과를 참조한다.

문제는 이 캐시가 차지하는 메모리가 컨텍스트 길이에 비례해서 커진다는 점이다. 그리고 기존 시스템은 이 메모리의 60~80%를 낭비하고 있었다.

| [*요청*], [*메모리 할당과 낭비*],  [요청 A: 컨텍스트 1,000토큰], [4,000토큰분 할당 → 3,000토큰분 낭비],
  [요청 B: 컨텍스트 3,000토큰], [4,000토큰분 할당 → 1,000토큰분 낭비],)

왜 이런 낭비가 발생하는가. 요청이 최종적으로 몇 토큰을 생성할지 미리 알 수 없기 때문이다. 시스템은 최대치를 기준으로 메모리를 미리 할당한다. 대부분의 요청은 최대치를 쓰지 않는다. 남는 공간은 비어 있지만 다른 요청이 쓸 수 없다. 60~80% 낭비의 원인이 여기에 있다.

## vLLM과 PagedAttention: OS에서 답을 찾다

2023년, UC Berkeley의 Woosuk Kwon 팀이 이 문제를 해결하는 시스템을 발표했다. vLLM이라는 이름의 이 시스템은 SOSP(Symposium on Operating Systems Principles, 운영체제 원리 심포지엄) 2023에서 발표됐다. 운영체제 학회에서 AI 시스템이 발표된 것이 의미심장하다. 해법이 OS에서 왔기 때문이다.

아이디어는 이렇다. 운영체제는 프로그램의 메모리를 페이지(page)라는 고정 크기 블록으로 나눠서 관리한다. 프로그램이 연속된 메모리를 요구하지만, OS는 물리적으로는 비연속적인 페이지들을 이어붙여서 연속인 것처럼 보이게 한다. 이것이 가상 메모리(virtual memory)다.

vLLM은 이 가상 메모리 기법을 KV-Cache에 적용했다. PagedAttention(페이지드 어텐션)이라는 이름의 이 기법은 KV-Cache를 고정 크기 페이지로 나눠서 관리한다. 연속된 큰 블록을 미리 할당하는 대신, 필요할 때마다 페이지 단위로 할당한다.

| [*지표*], [*기존 시스템*], [*vLLM (PagedAttention)*],  [KV-Cache 메모리 낭비], [60~80%], [4% 미만],
  [처리량 (동일 지연 시간)], [기준], [2~4배],
  [GPU 활용률 (고동시성)], [40~60%], [85~92%],)

이 숫자의 의미를 풀어보자. 메모리 낭비가 60~80%에서 4% 미만으로 줄었다는 것은, 같은 GPU로 2배 이상의 요청을 동시에 처리할 수 있다는 뜻이다. 처리량이 2~4배 늘었다는 것은, 같은 하드웨어 비용으로 2~4배 많은 사용자를 서비스할 수 있다는 뜻이다. GPU가 가장 비싼 자원인 AI 서빙에서, 같은 GPU로 2~4배 효율을 내는 것은 비용을 절반 이하로 줄이는 것과 같다.

> **핵심**: vLLM의 PagedAttention은 AI 추론의 비용 구조를 바꿨다. 해법은 새로운 AI 알고리즘이 아니라 1960년대부터 있던 OS의 가상 메모리 기법이었다.

## vLLM V1: 2025년의 대규모 업그레이드

2025년 1월, vLLM V1이 출시됐다. 이전 버전 대비 1.7배 속도가 향상됐다. 세 가지 핵심 개선이 있었다.

첫째, 제로 오버헤드 프리픽스 캐싱(zero-overhead prefix caching)이다. 프리픽스 캐싱이란 여러 요청이 같은 시스템 프롬프트로 시작할 때, 그 공통 부분의 KV-Cache를 재사용하는 기법이다. PART III에서 *안정적인 것을 앞에, 변하는 것을 뒤에 배치하라*고 했던 것이 여기서 비용 절감으로 직접 이어진다. 시스템 프롬프트가 10K 토큰이고 모든 요청에 공통이라면, 프리픽스 캐싱은 매 요청마다 10K 토큰분의 연산을 절약한다.

둘째, 추측 디코딩(Speculative Decoding)이다. 작은 모델이 답변의 초안을 빠르게 생성하고, 큰 모델이 그 초안을 검증한다. 초안의 대부분이 채택되면 전체 속도가 올라간다. 시험 답안을 빠르게 초안 작성하고, 꼼꼼하게 검토하는 것과 같다.

셋째, 분리형 프리필/디코드(Disaggregated Prefill/Decode)다. 입력을 처리하는 단계(프리필)와 토큰을 생성하는 단계(디코드)를 물리적으로 분리한다. 두 단계의 연산 특성이 다르기 때문이다. 프리필은 대량의 토큰을 한꺼번에 처리하므로 연산 집약적이고, 디코드는 토큰을 하나씩 생성하므로 메모리 집약적이다. 둘을 분리하면 각각에 최적화된 하드웨어를 배정할 수 있다.

2026년 현재 vLLM은 LLM 추론 서빙의 사실상 표준이다.

![KV-Cache의 메모리 구조. PagedAttention이 메모리 낭비를 60~80%에서 4% 미만으로 줄인 원리를 보여준다.](/assets/consulting-cropped/20-kvcache.png)

이 그림에서 주목할 것은 기존 방식과 PagedAttention 방식의 메모리 할당 패턴 차이다. 기존 방식은 큰 연속 블록을 할당하고 대부분을 비워두지만, PagedAttention은 필요한 페이지만 할당한다.

## 프로덕션 서빙 아키텍처의 구조

가장 단순한 서빙 구성은 사용자 요청이 API 서버를 거쳐 vLLM이 돌아가는 GPU 서버에 도달하고, 응답이 돌아오는 구조다. 프로덕션 환경에서는 이것이 복잡해진다.

| [*구성 요소*], [*역할*],  [로드밸런서], [요청을 여러 API 서버에 분산],
  [API 서버 (복수)], [인증, 요율 제한, 요청 전처리],
  [vLLM 클러스터 (GPU 복수)], [모델 추론 실행, KV-Cache 관리],
  [프리픽스 캐시], [공통 시스템 프롬프트의 KV-Cache 재사용],)

이 구조에서 프리픽스 캐시가 비용에 큰 영향을 준다. 같은 시스템 프롬프트로 시작하는 요청이 반복되면 해당 부분의 KV-Cache를 재사용한다. PART III에서 다뤘던 컨텍스트 배치 원칙이 여기서 실질적 비용 차이를 만드는 셈이다.

## GPU 독점은 언제까지 지속되는가

2026년 현재 AI 추론의 대부분은 NVIDIA GPU에서 돌아간다. H100, H200이 프로덕션 환경의 표준이다. 그러나 대안이 등장하고 있다.

| [*하드웨어*], [*개발사*], [*특성*],  [LPU], [Groq], [추론 전용 칩. 지연 시간이 GPU 대비 수배~수십배 빠르다],
  [TPU v6], [Google], [학습과 추론 모두 지원. Google Cloud 전용],
  [Trainium/Inferentia], [AWS], [AWS 전용 칩. 자체 클라우드 내 최적화],)

왜 전용 칩이 나오는가. GPU는 원래 그래픽 처리를 위해 설계된 칩이다. AI 추론에 효율적으로 작동하지만, AI 추론만을 위해 최적화된 것은 아니다. 전용 칩은 불필요한 회로를 제거하고 AI 추론에 필요한 연산만 남겨서 효율을 극대화한다. 범용 트럭과 전용 배달차의 차이와 같다. 범용 트럭은 무엇이든 실을 수 있지만, 배달만 하려면 전용 배달차가 더 효율적이다.

그러나 2026년 현재 NVIDIA의 생태계가 압도적으로 강하다. CUDA라는 프로그래밍 환경, 수만 개의 최적화된 라이브러리, 10년 이상 축적된 개발자 커뮤니티가 진입 장벽을 만들고 있다. 하드웨어 성능만으로는 생태계를 이기기 어렵다.

## 컨텍스트 엔지니어링이 곧 비용 엔지니어링이다

이 장의 내용이 왜 모델을 서빙하지 않는 사람에게도 중요한가. 이유는 단순하다. 컨텍스트 길이가 API 가격에 직접 반영되기 때문이다.

컨텍스트 10K 토큰으로 질문하는 것과 100K 토큰으로 질문하는 것의 비용 차이는 10배가 아니다. KV-Cache 비용이 비선형으로 증가하므로 10배 이상이 될 수 있다. 같은 질문에 같은 답을 얻으면서 컨텍스트를 절반으로 줄일 수 있다면, 비용도 절반 이하로 줄어든다.

PART III에서 다뤘던 컨텍스트 엔지니어링의 원칙들, 필요한 정보만 넣고 불필요한 것은 압축하고 안정적인 것을 앞에 배치하는 것, 이 모든 것이 성능만이 아니라 비용도 최적화한다. 컨텍스트 엔지니어링은 성능 엔지니어링인 동시에 비용 엔지니어링이다.

> **핵심**: 추론 인프라를 이해하면 비용을 통제할 수 있다. vLLM의 PagedAttention이 서빙 비용을 2~4배 줄였듯이, 컨텍스트 엔지니어링은 사용자 측에서 비용을 줄이는 방법이다. 모델 성능과 비용 효율은 분리할 수 없다.
