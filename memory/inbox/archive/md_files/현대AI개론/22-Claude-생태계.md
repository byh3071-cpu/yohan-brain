# 5개 계층이 하나의 생태계에서 만난다

*이 책의 PART III에서 다뤘던 프롬프트, 컨텍스트 엔지니어링, 하네스, MCP, Skills의 5개 계층은 추상적 개념이 아니다. Claude 생태계에서 이 다섯 계층이 구체적 제품으로 구현되어 있다.*

## 안전이 제품 경쟁력이 되는 회사

Anthropic이 다른 AI 회사와 다른 지점은 하나다. 대부분의 AI 회사가 *먼저 만들고 나중에 안전*이라면, Anthropic은 *안전이 곧 제품 경쟁력*이라는 입장을 취한다.

이것이 왜 중요한지를 앞 글들과 연결하면 명확해진다. 환각 감소, 가드레일 품질, 프롬프트 인젝션 방어, 이 모든 것이 안전 연구에서 나왔다. 헌법 AI(Constitutional AI)는 인간 피드백 없이도 AI가 스스로 행동을 교정하게 하는 기법이다. RLHF(Reinforcement Learning from Human Feedback, 인간 피드백 기반 강화학습)와 RLAIF(Reinforcement Learning from AI Feedback, AI 피드백 기반 강화학습)는 모델의 출력을 인간 또는 AI의 피드백으로 정렬하는 학습 방법이다. 해석가능성 연구(Mechanistic Interpretability)는 모델 내부에서 실제로 무엇이 일어나는지를 분석하는 분야다.

이것들이 논문에 머무르는 것이 아니다. Claude가 다른 모델보다 덜 환각하고 지시를 더 잘 따르는 이유 중 하나가 이 안전 연구의 직접적 반영이다. 안전과 성능이 트레이드오프가 아니라 같은 방향을 가리키는 셈이다.

## 5개 계층이 Claude에서 어떻게 매핑되는가

PART III에서 정리한 AI 활용의 5개 계층 구조가 Claude 생태계에서 구체적으로 구현된다.

| [*계층*], [*개념*], [*Claude 구현*],  [1. 프롬프트], [사용자의 지시], [Claude.ai 채팅, API messages],
  [2. 컨텍스트 엔지니어링], [정보 설계], [Projects, Memory Tool, CLAUDE.md],
  [3. 하네스], [실행 환경], [Claude Code (CLI, Desktop, Web, IDE)],
  [4. MCP], [외부 연결], [MCP 서버 생태계],
  [5. Skills], [전문성 패키지], [.claude/skills/ + 플러그인],)

이 표의 핵심은 각 행이 독립적 제품이 아니라 *하나의 진화 축 위에 놓여 있다*는 점이다. 1층 위에 2층이 쌓이고, 2층 위에 3층이 쌓인다. 프롬프트 없이는 컨텍스트 엔지니어링이 불가능하고, 컨텍스트 엔지니어링 없이는 하네스가 제대로 작동하지 않는다. 이 계층 구조를 이해하면 Claude 생태계 전체가 하나의 그림으로 보인다.

## Claude Code가 코딩 도구에서 범용 에이전트로 진화한 이유

앞 글에서 Claude Code를 코드 에이전트로 소개했다. 여기서 한 걸음 더 들어간다. Claude Code는 코딩 도구가 아니라 범용 에이전트 인프라로 진화하고 있다.

2025년 9월, Claude Code SDK가 Claude Agent SDK로 이름이 바뀌었다. 이름 변경의 이유가 명확하다. 이 하네스가 코딩뿐 아니라 거의 모든 주요 에이전트 루프를 구동하게 됐기 때문이다. Anthropic 자체가 Claude Code를 *에이전트 하네스*라고 부른다.

PART III에서 하네스의 설계 철학을 다뤘다. *비즈니스 로직을 최소화하고 모델이 대부분의 일을 한다.* Claude Code는 이 철학의 대표적 구현이다. 하네스 자체는 가볍고, 모델의 능력에 의존한다. 이 설계가 범용성을 가능하게 한 것이다. 코딩에 특화된 로직이 아니라, 에이전트 루프라는 범용 구조 위에 서 있기 때문이다.

사용 가능한 형태는 네 가지다. CLI(터미널), 데스크톱 앱(Mac/Windows), 웹 앱(claude.ai/code), IDE 확장(VS Code, JetBrains). 형태는 달라도 하네스는 동일하다.

![Claude 생태계의 4축 구조. 프롬프트에서 Skills까지 각 계층이 어떻게 연결되는지를 보여준다.](/assets/consulting-cropped/09-four-axes.png)

이 그림에서 주목할 것은 계층 간의 화살표다. 아래층이 위층을 지탱하는 구조이며, 한 층이 무너지면 그 위의 모든 층이 영향을 받는다.

## Claude 모델 계보: 2026년 4월 현재

Claude의 모델 라인업을 정리하면 다음과 같다.

| [*모델*], [*컨텍스트*], [*특성*],  [Opus 4.6], [1M 토큰], [최고 성능. 복잡한 추론에 적합],
  [Sonnet 4.6], [1M 토큰], [Opus급 성능에 더 빠른 출력 속도],
  [Sonnet 4.5], [200K 토큰], [성능과 속도의 균형],
  [Haiku 4.5], [200K 토큰], [가장 빠르고 가장 저렴],)

Opus 4.6과 Sonnet 4.6이 1M 토큰 컨텍스트를 갖고 있다. PART III에서 *컨텍스트 윈도우가 크다고 다 채우면 안 된다*고 했지만, 넓은 공간이 있다는 것 자체가 컨텍스트 엔지니어링의 자유도를 높인다. 100K 토큰이 한계인 모델에서는 선택해야 했던 정보를, 1M 토큰 모델에서는 함께 넣을 수 있다.

Haiku 4.5는 비용에 민감한 대량 처리에 적합하다. 분류, 추출, 간단한 요약처럼 고난도 추론이 필요 없는 작업에서 Opus를 쓰는 것은 낭비다. 작업의 난이도에 맞는 모델을 선택하는 것이 비용 효율의 핵심이다. 앞 글에서 다룬 추론 인프라의 비용 구조를 모델 선택에 적용하는 셈이다.

## MCP와 Skills가 만나는 지점

PART IV에서 MCP와 Skills를 각각 다뤘다. Claude 생태계에서는 이 둘이 함께 작동한다. 구체적 예를 보자.

*이 이슈를 보고 PR을 만들어줘*라고 요청하면, 세 계층이 동시에 작동한다. MCP가 GitHub에 접근해서 이슈를 가져오고 PR 생성 API를 호출한다. Skills가 *이 프로젝트에서 PR은 이런 형식이어야 하고, 리뷰어는 최소 2명이며, 테스트 커버리지는 80% 이상이어야 한다*는 규칙을 적용한다. 하네스가 이 전체를 중개하고, 컨텍스트를 관리하며, 검증 루프를 돌린다.

| [*계층*], [*역할*], [*비유*],  [MCP], [외부 서비스 접근 (GitHub, Slack, DB)], [손과 발],
  [Skills], [내부 전문성 (규칙, 형식, 품질 기준)], [전문 지식],
  [하네스], [통합 관리 (컨텍스트, 도구 호출, 검증)], [두뇌],)

이 세 계층이 Claude에서 가장 깊게 통합되어 있다. 이유는 단순하다. MCP와 Skills 둘 다 Anthropic이 설계했기 때문이다. 다른 플랫폼도 MCP를 도입했지만, Skills까지 통합한 것은 Claude 생태계가 유일하다.

> **핵심**: MCP는 외부 세계에 대한 접근이고, Skills는 내부 전문성이다. 이 둘이 하네스 위에서 만날 때 에이전트는 단순 도구에서 전문가로 진화한다.

## Claude를 제대로 쓰는 다섯 가지 원칙

이 책에서 다뤄온 개념들을 Claude 생태계에서의 실천으로 압축하면 다섯 가지가 된다.

첫째, CLAUDE.md를 잘 쓴다. 프로젝트의 규칙, 구조, 금지 사항을 여기에 명시한다. 이 파일이 매 세션마다 자동 로드된다. 컨텍스트 엔지니어링의 Write 축을 실현하는 가장 직접적인 방법이다.

둘째, 세션을 짧게 끊는다. 컨텍스트 사용률이 70~80%를 넘으면 새 세션을 시작한다. PART III에서 다뤘듯이 컨텍스트가 가득 차면 모델의 주의력이 분산되고 품질이 떨어진다.

셋째, MCP는 최소한으로 유지한다. 필요한 외부 서비스만 연결하고, 서버당 도구 5개 이하를 유지한다. 도구가 너무 많으면 Context Confusion이 발생한다. 모델이 어떤 도구를 써야 할지 혼란스러워하는 현상이다.

넷째, Skills를 만든다. 프로젝트에 반복되는 절차가 있으면 스킬로 정의한다. PR 형식, 코드 리뷰 기준, 배포 절차 같은 것들이다. 한 번 정의하면 이후 모든 세션에서 일관되게 적용된다.

다섯째, 서브에이전트를 활용한다. 탐색 작업은 메인 컨텍스트에서 분리해서 서브에이전트에 맡긴다. 읽기 전용으로 격리하면 메인 컨텍스트가 오염되지 않는다. PART III의 Isolate 축을 실현하는 방법이다.

## 생태계 수준에서 본 Claude의 위치

개별 제품 비교를 넘어 생태계 수준에서 보면, Claude의 위치가 더 명확해진다.

| [*계층*], [*Claude*], [*OpenAI*], [*Google*],  [모델], [Opus/Sonnet/Haiku], [GPT-4o/o3], [Gemini 2.5],
  [하네스], [Claude Code (Agent SDK)], [Codex, ChatGPT], [Gemini Code Assist],
  [프로토콜], [MCP (개방형)], [자체 도구 호출], [자체 도구 호출],
  [전문성 패키지], [Skills], [Custom GPTs], [Gems],)

이 표에서 주목할 것은 프로토콜 행이다. MCP는 개방형 프로토콜이다. Anthropic이 만들었지만 Claude에만 묶여 있지 않다. OpenAI, Google 등 다른 플랫폼도 MCP를 도입할 수 있다. 실제로 2025~2026년에 걸쳐 여러 플랫폼이 MCP 지원을 추가하고 있다. 이것은 Anthropic의 전략적 선택이다. 프로토콜을 개방함으로써 생태계의 중심에 서려는 것이다.

USB가 특정 회사의 제품이 아니라 범용 표준이 된 것처럼, MCP가 AI 도구 연결의 범용 표준이 되면 그 표준을 설계한 Anthropic이 생태계의 중심에 위치하게 된다. 제품이 아니라 표준으로 경쟁하는 전략인 셈이다.

> **핵심**: Claude 생태계의 차별점은 단일 제품의 성능이 아니라, 5개 계층이 일관된 설계 철학 아래 통합되어 있다는 점이다. MCP라는 개방형 프로토콜로 외부 생태계까지 연결하면서, 폐쇄와 개방 사이의 독특한 위치를 점유하고 있다.

이것으로 PART VI을 마친다. 오픈소스 LLM, 추론 인프라, 코드 어시스턴트, Claude 생태계를 거치면서 AI가 실제로 작동하는 기반 구조를 살펴봤다. 기술의 원리를 아는 것과 그 기술이 돌아가는 인프라를 이해하는 것은 별개의 역량이다. 둘 다 갖춰야 AI를 제대로 쓸 수 있다.
