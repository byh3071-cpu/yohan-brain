# 원문 → 요약 프로토콜 (로컬판)

> **트리거:** 새 원본 자료(아티클·영상·녹음·스크린샷·URL·텍스트)가 생겼을 때 · "자료 정리/요약해줘" 요청 시.
> 전체 구조·폴더 역할은 `docs/KNOWLEDGE-LOOP.md` 참조. 노션 원본 프로토콜과 동일한 파이프라인을 로컬 경로로 실행한다.

## 통합 파이프라인

```
[1] 원문→요약 (Step 0~4)
    캡처 → RESOURCE 등록 → SUMMARY 작성 → 상태 업데이트
    Step 4.5: 교차검증 + 승격 + 2차보강
    │
    ▼ 🔀 체크포인트 1 — "역전파 필요한 문서 있어? 진행할까?"
[2] 역전파 (Step 4.6)
    범위 결정 → 대상 탐색 → 보강 리스트 → 실행
    │
    ▼ 🔀 체크포인트 2 — "온톨로지 추출 진행할까?"
[3] 온톨로지 추출 (Step 4.7)
    개체 식별 → 트리플 추출 → 사전·키워드·인물 반영 → 트리플 맵 등록
    │
    ▼ 복귀
Step 5~6: 인박스 아카이브 → 기록 (세션 로그·결정)
```

**체크포인트 규칙:** 스킵 가능, 생략 불가 — 각 체크포인트에서 반드시 진행/스킵을 판단·보고한다. 단순 메모·짧은 요약이면 "스킵 추천" 멘트, 기존 허브·사전과 직접 관련이면 "진행 추천" 멘트. 사용자가 "전체 문서 보강" 요청 시 묻지 않고 실행.

## 입력 타입별 분기 (Step 1·2 변형)

**공통 대원칙: 확보된 원문 텍스트에서만 요약한다.** 링크 제목·URL·썸네일만 보고 추측 요약 금지. 원문 확보 실패 = 요약 보류 → `memory/inbox/automation-review.md`에 사유와 함께 등록하고 사람/에이전트 세션에서 수동 확보 후 Step 2 진행.

| # | 입력 | 원문 확보 (자동) | 확보 실패 시 폴백 |
|---|------|-----------------|------------------|
| 1 | 텍스트·긴 메모 | 봇 → `inbox/telegram/` append (텍스트 자체가 원문) | — |
| 2 | 이미지·스크린샷 | 봇 OCR(tesseract) → batch 정제 (`notion-ocr-pipeline.md`) | OCR 품질 C등급 → review 큐, 원본 이미지는 노션 보존 |
| 3 | 일반 아티클 URL | `ingest_url` (Readability 본문 추출) → `ingest/url/` | 본문 추출 실패 문구 확인 → 에이전트 세션에서 직접 열람·확보 |
| 4 | GitHub 레포 | batch `ingestGithubAndUpsertCard` (README 기반 why-how 카드) | README 부실 → 코드 구조 직접 훑고 `[추론]` 표기 |
| 5 | YouTube | `ingest_url` (oembed 제목 + 자막 추출, `has_transcript`) | 자막 없음 → 트랜스크립트 수동 확보, 없으면 요약 보류 |
| 6 | 네이버 블로그 | ❌ 자동 불가 (iframe 구조로 Readability 실패) | `m.blog.naver.com` 모바일 URL 재시도 → 실패 시 본문 복붙 or 스크린샷→OCR(#2 경로) |
| 7 | Threads·X·인스타 등 SNS | ❌ 자동 불가 (로그인 장벽·JS 렌더링) | **보낼 때 본문 텍스트를 링크와 함께 복붙** (아래 캡처 규칙) |
| 8 | 기타 영상 (비메오·네이버TV·릴스 등) | ❌ 자막 도구 없음 | 시청 메모 or 트랜스크립트 확보 후 요약 — 없으면 요약 보류 |
| 9 | RSS | `ingest:all` → `ingest/rss/` 자동 수집 | 가치 있는 것만 Step 2 진입 (전수 요약 아님) |

### 타입별 RESOURCE·SUMMARY 특칙

- **#1 텍스트:** inbox 블록이 RESOURCE. 짧으면 축소 포맷, 처리 후 archive (Step 5).
- **#3 아티클:** RESOURCE는 메타+추출 본문(원칙상 링크 우선이나 추출본은 보존). 요약은 추출 본문 기준 — 추출이 부분적이면 `[부분 읽기]` 태그.
- **#4 GitHub:** 표준 요약 구조 대신 **why-how 카드 구조** — 무엇인가 / 왜 쓰나(기존 도구 대비) / 어떻게 붙나(Yohan OS 적용점) / 도입 판단. 스타 수·벤치마크 등 수치는 검증 불가 시 `[미검증]`.
- **#5 YouTube:** 요약은 자막 기준, 긴 영상은 타임라인(주제 구간) 구조 권장. 자막 자동생성 오인식 가능 — 고유명사는 교차 확인.
- **#6·#7 블로그·SNS:** 포스트는 삭제·비공개 전환이 잦다 → **"외부에서 사라질 수 있는 원문"으로 간주, 확보된 본문은 전문 보존** (Step 1 예외 조항 적용). SNS는 작성자·날짜·스레드 맥락(몇 번째 답글인지)을 메타로 기록.
- **#8 영상:** 요약 출처가 "시청 메모"면 frontmatter에 `source_basis: memo` 명시 — 자막 기반과 신뢰도 구분 (온톨로지 신뢰도: 자막=3, 메모=2).

### 텔레그램 캡처 규칙 (사람 쪽 1가지)

자동 추출 불가 소스(#6·#7·#8)는 **링크만 보내면 메타만 남고 요약이 보류된다.** 캡처 시점에 본문 텍스트(또는 스크린샷)를 같이 보내는 것이 유일한 확실한 원문 확보다 — "본문 복붙 한 번 > 나중에 죽은 링크".

## Step 0-Pre. 사전 진단 (긴 원본일 때)

원본 로드 직후 보고: 읽기 % · 등급(A: 90%+, B: 70~89%, C: 70% 미만) · 권장 조치(그대로 진행/분할 읽기/보충 요청). C등급으로 진행 시 SUMMARY에 `[부분 읽기]` 태그.

## Step 1. RESOURCE 등록 (원본 보존)

- 위치: `memory/ingest/` 하위 (url/, rss/, 기타 raw)
- 원칙: **원본 전문 복제 금지** — 메타데이터(출처 URL·수집일·핵심 태그) + 링크가 기본. 외부에서 사라질 수 있는 원문만 전문 보존.

## Step 2. SUMMARY 작성 (정제 요약)

- 위치: `memory/ingest/insights/{kebab-case-id}.md`
- 템플릿: `memory/templates/insight-template.md` (frontmatter: id·date·domain·tags·related·status)
- **필수:** frontmatter `related`에 원본(RESOURCE) 경로 또는 출처 표기 — 원본 미연결 SUMMARY 금지.
- (선택) 학습 의도 한 줄 — "이걸 왜 보는지"가 있으면 본문 최상단에 `[학습 의도] ...` 기재. 없으면 기존 트리플·위키 맥락으로 추론해 "나에게 왜 중요한가"를 포함.

### 표준 요약 구조

```markdown
## 핵심 요약 (3줄)
## 핵심 키워드
## 본문 — 논지 전개      ← 긴 원문 필수: 사례·수치·논증 구조를 절 단위로 (3줄 요약의 근거가 다 들어가야 함)
## 댓글·반응 요약 (있으면) ← 수렴 vs 반박으로 나눠 정리 — 원문에 없는 반론이 여기서 나옴
## 내 생각               ← 긴 원문 필수: 설득되는 부분 / 경계할 부분(저자 편향·논리 급소) / 실전 결론
## 인사이트 → 적용
## Yohan OS 적용
- 바로 적용 가능한 것: / 도입하면 좋은 것: (해당 없으면 "없음")
## 트리플 맵        ← Step 4.7에서 추가
## 소스 출처
```

- 원본에 이미 AI 요약이 있으면 **삭제·축약 없이 이관 후** 포맷만 보강.
- 짧은 원본(한 문단 이하)은 축소 포맷(핵심 요약+적용만). **긴 원문(아티클·영상)을 3줄 요약+적용만으로 끝내지 않는다** — 본문 전개·반응·내 생각이 빠지면 원문을 다시 읽어야 하므로 SUMMARY의 존재 이유가 사라짐. [2026-06-12 사용자 피드백]
- **인물 감지 훅:** 저자·발화자·핵심 등장인물 감지 시 → `memory/wiki/entities/`에서 검색 → 있으면 `related`에 연결, 없으면 "인물 등록할까?" 1회 제안 (등록 시 `entity_type: person`). 1회성 인용은 제외.
  - 자율 세션(사용자가 실시간 응답 불가)에서는: **반복 등장이 명확**하면(예: ingest에 동일 저자 원본 다수 보유) 등록 실행 후 보고, 애매하면 등록 보류 + 후보로 보고. [2026-06-12 실전 1회전에서 보완]

## Step 4.5. 교차검증 + 승격 + 2차보강

**A. 교차검증** — 같은 domain·tags의 기존 insights 2개 이상이면 수렴·충돌 대조. `triple-map.md`에서 핵심 키워드 Subject 매칭 → 같은 Subject에 다른 Relation이 붙어 있으면 관점 충돌/보완 발견 → SUMMARY에 반영.

**B. 승격 체크** — 승격 후보를 리스트업해 사용자 확인 (모든 SUMMARY가 승격 대상은 아님):

| 승격 대상 | 기준 | 방법 |
|----------|------|------|
| `memory/wiki/` (= AI 사전·스택 사전·인물 DB) | 재사용할 개념·도구·기술·인물 | MCP `promote_to_wiki` 또는 `npm run promote-wiki` |
| `memory/knowledge-hub/` | 여러 소스를 종합할 주제 문서 | `knowledge-hub/TEMPLATE.md`로 생성 또는 기존 문서 보강 |
| `memory/knowledge-hub/keywords.md` | **프롬프트에 넣으면 출력이 달라지는** 키워드만 | 표에 행 추가 |
| `memory/rules/` · `memory/profile.yaml` | 새 절차·원칙·취향 | 해당 파일 갱신 |

**C. 2차보강** — 누락 핵심 내용·연결성 점검. 수치·날짜·고유명사는 웹 검색 교차 확인, 불가 시 `[미검증]` 표기. 긴 원본은 ① 목표 커버리지 ② 도구·스킬명 완전성 ③ 후반부 데모 결과 누락 3종 체크.

## Step 4.6. 역전파 (Knowledge Backfill) — 체크포인트 1

새 지식 1건을 기존 문서 전체에 전파한다.

1. **범위 결정** — 허브(knowledge-hub·rules·docs 스펙) / 하위(wiki·관련 insights) / 전체 / 스킵
2. **대상 탐색 + 대조** — 관련 wiki 카드, knowledge-hub 문서, rules, AGENTS.md·docs 스펙에서 체크: 구버전 정보 / 누락 정보 / 연결(related) 누락 / 맥락 충돌
3. **보강 리스트업** — `| 문서 | 보강 필요 | 이유 |` 대조표 제시 → 사용자 확인 (없으면 "역전파 완료, 보강 불필요" 선언)
4. **실행** — 기존 구조·스타일 유지, 보강 끝에 `[YYYY-MM-DD 역전파]` 태그. 전수 조사 아님 — 관련성 높은 문서만.

## Step 4.7. 온톨로지 추출 (Triple Extraction) — 체크포인트 2

**도메인 무관 원칙:** 기술/비기술(비즈니스·철학·감정·자기이해) 구분 없이 의미 관계가 있으면 추출한다.

1. **개체 식별 (5~10개)** — 개념/인물/도구/조직. 핵심만, 전부 뽑지 않기.
2. **트리플 추출 (3~7개)** — `[주어] --관계--> [목적어]`. 표준 Relation 팔레트는 `knowledge-hub/triple-map.md` 상단 참조. 새 인사이트가 담긴 관계 위주, 불확실하면 `related_to`.
3. **기존 트리플 대조** — `triple-map.md`에서 Subject 검색:
   - Subject+Relation 동일 → 중복, 등록 스킵
   - Subject 동일+Relation 상이 → 🔥 관점 확장/충돌 — 등록 + SUMMARY 교차검증에 메모
   - Subject 신규 → 정상 등록
4. **사전 반영** — 신규 개념 → wiki 승격 후보로 제안(Cap 고려해 사용자 확인). 양 개체가 모두 wiki에 있으면 `related_*` 상호 연결.
5. **키워드 스캔** — 키워드 DB 기준(프롬프트 영향 키워드 전용)으로 후보 식별. **결과를 반드시 한 줄 보고:** "키워드: N건 등록" 또는 "키워드: 스킵 (이유)" — 침묵 금지.
6. **등록** — SUMMARY 본문에 `## 트리플 맵` 섹션 추가 **그리고** `triple-map.md`에 행 추가 (둘 다, 별개).
7. **신뢰도 자동 부여** — 공식문서=5 / 전문가 강의=4 / 분석글=3 / 추론=2 / 인상=1
8. **판단 근거 기록 (선택)** — 새 인사이트·전략 판단이 있으면 근거 트리플 2~3개 + 신뢰도를 `### 판단 근거`로 명시 (해석 가능성).

## Step 5. 인박스 아카이브

처리 완료된 inbox 블록은 `telegram-inbox.md` 규칙대로 archive 이동 또는 삭제 — **inbox는 비우는 곳**.

## Step 6. 기록 + 동기화

- 파이프라인 전체를 1건으로 `memory/logs/sessions/`에 기록 (서브 프로토콜 실행/스킵 여부 포함). 실패·부분성공도 기록 생략 금지.
- 구조적 결정이 있었으면 `memory/decisions/` (MCP `append_decision`).
- wiki 변경이 있었으면 `memory/wiki/log.md` append + `index.md` 갱신.
- 노션 반영 필요 시 MCP `sync_to_notion` (SoT Key 멱등).
