# awesome-design-md — 왜 쓰는지 · 어떻게 쓰는지

## 정의 (1~2문장)

- 여러 서비스 UI 스타일을 **`DESIGN.md` 텍스트 규격**으로 모아, 코딩 에이전트가 프로젝트 화면 톤을 빠르게 맞추게 하는 **오픈 레퍼런스 컬렉션**이다.
- Figma 없이 “이 앱처럼”을 **색·타이포·컴포넌트 규칙 문서**로 고정해, 프롬프트마다 디자인이 흔들리는 문제를 줄인다.

## Verified (소스 기반)
- **GitHub:** [VoltAgent/awesome-design-md](https://github.com/VoltAgent/awesome-design-md) [source: awesome-design-md-github-why-how]
- **README 인제스트:** `memory/ingest/url/url-f9df08c201fdc4c7.md` — DESIGN.md 구조 섹션·사이트별 컬렉션 목록·라이선스 문구는 원문 기준으로 본다. [source: awesome-design-md-github-why-how]
- 다양한 서비스의 UI 스타일을 `DESIGN.md` 텍스트 규격으로 모아, 코딩 에이전트가 프로젝트 화면 톤을 빠르게 맞추게 돕는 레퍼런스 모음 레포다. [source: awesome-design-md-github-why-how]
- “이 앱처럼 보이게 만들어 줘”를 감각으로 설명하지 않고, 색·타이포·컴포넌트 규칙을 글로 적은 파일(`DESIGN.md`)을 넣어 에이전트가 동일한 스타일로 코드를 생성하게 하는 방식이다. [source: awesome-design-md-github-why-how]
- **내 일:** 랜딩/대시보드 UI를 빠르게 만들 때 Figma 없이도 `DESIGN.md` 한 파일로 시안 스타일을 고정해, 프롬프트마다 디자인 톤이 흔들리는 문제를 줄일 수 있다. [source: awesome-design-md-github-why-how]
- 실제 제품 수준에서 이 방식이 디자인 시스템 토큰(컴포넌트 라이브러리·테마 코드)과 얼마나 안정적으로 연결되는지, 프로젝트 규모가 커질수록 보완 규칙이 필요해 보인다. [source: awesome-design-md-github-why-how]
